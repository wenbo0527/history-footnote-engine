# 分析报告：高自由度RPG引擎中AI与事件总线的边界划分

## 核心判断原则

**Dex Horthy（HumanLayer创始人）的判断标准**：真正优秀的Agent，本质大多是确定性代码，只是在关键节点巧妙地“撒”上LLM调用。

**转化为RPG引擎的判断标准**：

| 判断维度 | 交给事件总线（确定性代码） | 交给AI（LLM） |
|---------|----------------------|-------------|
| **可枚举性** | 输入输出可穷举 → 总线 | 输入输出开放、需创造性 → AI |
| **失败代价** | 失败不可接受（游戏崩溃/逻辑矛盾） → 总线 | 失败可容忍（对话不自然/建议不完美） → AI |
| **一致性要求** | 必须100%一致 → 总线 | 允许每次不同 → AI |
| **验证可能性** | 可用if/switch验证 → 总线 | 需要语义理解才能验证 → AI |

## 一、必须走事件总线（确定性代码）的部分

### 1.1 状态管理——100%确定性

| 状态类型 | 为什么不能交给AI | 实现方式 |
|---------|---------------|---------|
| **世界状态**（时间、地点、天气） | AI会遗忘或矛盾 | StateGraph节点，代码读写 |
| **任务状态**（未开始/进行中/已完成） | AI可能重复触发或跳过 | 事件驱动+状态机 |
| **角色关系值**（好感度、信任度） | AI无法精确计算数值 | 数值系统，代码维护 |
| **物品/背包** | AI会凭空生成或丢失物品 | 库存管理器，代码校验 |
| **历史锚点**（不可更改的历史事实） | AI可能违反历史约束 | 验证系统，代码拦截 |

**你的项目发现**：自主判断30%塌房率——这就是把状态管理交给AI的后果。

### 1.2 流程控制——100%确定性

| 流程类型 | 为什么不能交给AI | 实现方式 |
|---------|---------------|---------|
| **Agent调度顺序** | AI无法可靠决定“该轮到谁了” | LangGraph StateGraph，代码编排 |
| **对话轮次管理** | AI可能无限循环或提前结束 | 计数器+最大轮次限制 |
| **任务链推进** | AI可能跳过前置条件 | 前置条件检查，代码验证 |
| **存档/读档** | AI无法序列化状态 | 状态序列化，代码执行 |
| **错误重试** | AI会陷入死亡循环 | 确定性重试逻辑（最多3次） |

**12-Factor Agents的对应原则**：要素8“掌控你的控制流”——不要把核心控制流外包给黑箱。

### 1.3 规则执行——100%确定性

| 规则类型 | 为什么不能交给AI | 实现方式 |
|---------|---------------|---------|
| **战斗数值计算** | AI无法精确计算 | 公式系统，代码执行 |
| **物品使用规则** | AI可能允许不合理的组合 | 规则引擎，代码校验 |
| **历史事实验证** | AI可能生成与史实矛盾的内容 | PANGeA式验证系统，代码拦截 |
| **NPC行为边界** | AI可能让NPC做出不符合人格的事 | 大五人格参数+代码约束 |

**你的项目发现**：确定性逻辑必须交给代码——这就是规则执行不能交给AI的原因。

### 1.4 事件路由——100%确定性

| 路由类型 | 为什么不能交给AI | 实现方式 |
|---------|---------------|---------|
| **事件分发**（谁该收到这个事件） | AI可能遗漏订阅者 | 事件总线，发布-订阅模式 |
| **事件优先级** | AI无法可靠排序 | 优先级队列，代码执行 |
| **事件过滤**（哪些事件需要响应） | AI可能响应不该响应的 | 过滤器链，代码执行 |

## 二、应该交给AI（LLM）的部分

### 2.1 自然语言理解——AI的核心价值

| 场景 | 为什么必须AI | 实现方式 |
|-----|------------|---------|
| **玩家自由输入理解** | 无法枚举所有可能输入 | LLM解析→结构化JSON |
| **意图识别** | 玩家说“我想看看那个老头的底细”=调查NPC | LLM→意图分类+参数提取 |
| **模糊指令消歧** | “帮我对付他”是攻击还是说服？ | LLM→结合上下文判断 |

**12-Factor Agents的对应**：要素1“自然语言到工具调用”——LLM最不可替代的价值是将非结构化输入转化为结构化输出。

### 2.2 内容生成——AI的创造力价值

| 场景 | 为什么需要AI | 约束方式 |
|-----|------------|---------|
| **NPC对话生成** | 预写对话无法覆盖所有玩家行为 | 大五人格参数+历史锚点约束 |
| **环境描写** | 需要根据当前世界状态动态生成 | 场景模板+LLM填充细节 |
| **事件叙述** | “This will be remembered”后的后果叙述 | 历史锚点+LLM生成叙述 |
| **任务描述** | 同一任务在不同世界状态下描述不同 | 任务数据+LLM改写 |

### 2.3 Drama Manager的“判断”部分——AI的决策价值

| 场景 | 为什么需要AI | 确定性代码的配合 |
|-----|------------|---------------|
| **节奏感知** | 判断“故事是否拖沓”需要语义理解 | 代码提供玩家行为数据，AI判断节奏 |
| **NPC记忆回响** | 决定“何时让旧选择浮现”需要叙事直觉 | 代码维护记忆数据，AI决定浮现时机 |
| **角色均衡** | 判断“谁被冷落了”需要社交感知 | 代码追踪发言次数，AI创造介入机会 |

**关键**：Drama Manager是AI和确定性代码的混合体——代码提供数据，AI做判断，代码执行决策。

### 2.4 玩家建模——AI的分析价值

| 场景 | 为什么需要AI | 确定性代码的配合 |
|-----|------------|---------------|
| **行为模式识别** | “这个玩家偏好战斗还是对话”需要语义理解 | 代码记录行为日志，AI分析模式 |
| **情绪推断** | “玩家是否感到无聊/沮丧”需要情感理解 | 代码追踪行为指标（停留时间、选择速度），AI推断情绪 |

## 三、边界地带：AI生成+代码验证

这是最关键的设计模式——**AI负责生成，代码负责验证**。

### 3.1 PANGeA模式

```
玩家自由输入 → AI生成内容 → 验证系统（代码）检查 → 通过则执行 / 拒绝则重生成
```

### 3.2 具体应用场景

| 场景 | AI生成什么 | 代码验证什么 |
|-----|----------|------------|
| **NPC对话** | 对话文本 | 是否违反历史锚点、是否符合人格参数、是否超出叙事范围 |
| **事件触发** | 触发条件和后果 | 前置条件是否满足、后果是否在允许范围内 |
| **任务生成** | 任务描述和目标 | 目标是否可完成、奖励是否合理、是否与已有任务冲突 |
| **世界状态变更** | 变更提案 | 变更是否违反不可变规则（如历史锚点） |

### 3.3 “历史的注脚”的具体实现

```
┌──────────────────────────────────────────────────┐
│                  事件总线（确定性）                  │
│  状态管理 │ 流程控制 │ 规则执行 │ 事件路由 │ 验证拦截  │
└──────┬───────────────────────────────────┬───────┘
       │                                   │
       ▼                                   ▼
┌──────────────┐                  ┌──────────────┐
│  AI生成节点   │                  │  AI判断节点   │
│ NPC对话生成   │                  │ 节奏感知      │
│ 环境描写      │                  │ 玩家建模      │
│ 事件叙述      │                  │ NPC记忆回响   │
│ 意图理解      │                  │ 角色均衡      │
└──────────────┘                  └──────────────┘
       │                                   │
       ▼                                   ▼
   生成结果→验证系统                  判断结果→代码执行
  （代码拦截不合规内容）            （代码执行确定性动作）
```

| 角色数值系统 | **总线** | 数值计算不能有模糊空间 |
| 物品/背包管理 | **总线** | AI会凭空生成/丢失物品 |
| 历史锚点验证 | **总线** | 不可更改的事实必须代码守护 |
| Agent调度编排 | **总线** | LangGraph StateGraph=确定性DAG |
| 事件分发/路由 | **总线** | 发布-订阅必须可靠 |
| 存档/读档 | **总线** | 序列化必须确定性 |

## 五、与你的项目发现的对照

| 你的项目发现 | 12-Factor Agents的对应 | RPG引擎的对应 |
|------------|---------------------|-------------|
| 自主判断30%塌房率 | 要素8：掌控控制流 | 状态管理必须确定性 |
| 单Agent是多Agent的降维打击 | 要素10：小而专注的Agent | ORK的系统解耦 |
| 确定性逻辑必须交给代码 | 要素4：工具只是结构化输出 | 数据驱动=代码执行器 |
| LangGraph StateGraph架构 | 要素8：回归DAG | ORK的节点系统 |
| 历史锚点约束 | 要素9：压缩错误进上下文 | PANGeA验证系统 |

**核心结论**：你的项目实践和12-Factor Agents方法论、RPG引擎30年工程经验，三者指向同一个判断——**总线管骨架，AI管血肉**。骨架是确定性的，血肉是生成性的。骨架出错游戏崩溃，血肉出错最多不自然。
