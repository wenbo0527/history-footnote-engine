## 🔍 12 个 BUG 根因模式

### 模式 A: try/except 吞噬异常（最严重）

**症状**:代码抛错但测试/用户看不见
**统计**:后端 `except Exception` 出现 **42** 次

**典型案例**:
```python
# game_loop.py:295
except Exception as e:
    logger.exception("回合执行异常")  # ← 只 logger，不 raise
    print(f"[ERROR] 系统异常: {e}。继续下一回合。")
```

**为什么 chosen_voice/extract_narrative_node 没发现**:
1. Mock LLM 路径绕过 graph.invoke → 不触发这些函数
2. try/except 吞了真 LLM 路径的异常
3. 单测只看 narrative_history 数量,不验证内容正确性

**预防建议**:
- 关键路径(LLM 调用、narrative 解析)不直接 try/except
- 用 Sentry / 异常上报系统
- 测试覆盖异常重抛

### 模式 B: optional 字段直接渲染（最高频）

**症状**:字段 undefined 时显示 "undefined" 或空
**统计**:前端 `{#if...}` 6 处 + `archive.*` 引用 多处

**典型案例**:
```svelte
<!-- ArchiveCard (已修): {archive.character_name} -->
<!-- RecapModal.svelte: {item.current_date} (optional) -->
<!-- FateCardDetailModal: {card.use_hint} (optional) -->
```

**预防建议**:
- 所有 optional 字段用 `?.` + 兜底
- 派生函数模式:`deriveXxx()` 集中处理
- 写 BUG10/11 错误状态视觉提示

### 模式 C: 字段名/类名冲突

**症状**:后端 import 错类,行为不一致
**典型案例**:
```python
# game_loop.py:510
# GameEvent 在 dm_agent.tools 和 game_memory 各有定义
# 后端用错类,导致 event_log 不持久化
```

**预防建议**:
- 命名空间加前缀(避免 GameEvent 这种通用名)
- 同名类在 import 时 alias:`from x import Y as YX`
- 模块边界要明确

### 模式 D: 字段引用但不验证后端是否返回

**症状**:前端类型定义有 `markdown`,后端无
**典型案例**:`wiki.markdown` (CharacterWiki 类没此字段)
**统计**:`as any` 出现 **56** 次(类型不匹配绕过的源头)

**预防建议**:
- 后端先实现字段,前端再渲染
- 用契约测试(contract test)验证 API 响应
- 减少 `as any` 用法

### 模式 E: 字段类型不匹配（最隐蔽）

**症状**:GameState 类型定义与实际后端响应不一致
**典型案例**:
```svelte
<!-- ShareCardButton.svelte:54 -->
role: `${game.identity === 'weaving_male' ? '织工' : ...}·${game.character.name}`
<!-- 但 GameState 没有 identity 字段！应该是 game.character.identity -->
```

**预防建议**:
- 严格 TS 检查:`tsc --noEmit` 强约束
- 定期 review 字段引用
- 后端 mapper 单元测试

### 模式 F: 中文/英文单位混乱

**症状**:前端显示 `5.70 两`(机器精度)而不是 `5 两 7 钱`
**已修**:`currency.py` 统一单位

**预防建议**:
- 货币/日期/距离 等本地化数据集中在 `utils/`
- 不要 `.toFixed(2)` 后直接显示

### 模式 G: 状态可空但代码不检查

**症状**:`$game?.character.name` 后 `character.identity || character.occupation` 崩
**统计**:多处 `$game?.x` 后直接 `.x.y.z`

**预防建议**:
- TS `strictNullChecks: true`
- 用 optional chaining `?.` 全链
- 测试空状态(loading/error/empty)

### 模式 H: 旧字段保留"兼容"但实际已废弃

**症状**:`character_name` 等已废弃,保留是历史包袱
**统计**:Archive type 有 5 个 optional 旧字段

**预防建议**:
- 定期清理 deprecated 字段
- 后端 deprecation warning
- 前端不用旧字段

### 模式 I: 缺少集成测试

**症状**:单元测试通过,真 LLM/真游戏跑就崩
**已修**:本会话 W52 加了 20 回合 + 50 回合 + 真 LLM 1 回合

**预防建议**:
- 单元测试 + 集成测试 + 端到端 三层
- 集成测试用真 LLM(限定流)
- 至少 1 轮 end-to-end smoke

### 模式 J: 文案/数据无默认

**症状**:LLM 没生成 `description` 时显示空
**典型案例**:`WikiCharacter.description` (non-optional, 但后端可能空字符串)

**预防建议**:
- 所有显示文案有 default placeholder
- "暂无" / "—" 类兜底

### 模式 K: error handler 只 logger 不 propagate

**症状**:用户看不到错误,QA 不知道
**统计**:后端 18 处 `except Exception: logger.exception(...)` 后不 raise

**预防建议**:
- 顶层 error boundary (前端 Svelte 5)
- 关键路径必须 propagate
- CI 集成 `caplog` 检查

### 模式 L: 路由参数/查询参数未校验

**症状**:`/admin` URL 参数被脚本读取但未验证
**典型案例**:`params.get('admin') === 'true'` (adminMode.ts:21)

**预防建议**:
- 用 Zod / Yup 校验
- 集中路由参数解析
