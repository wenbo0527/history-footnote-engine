# v2.7+ DM Agent 性能分析报告

> **日期**：2026-07-09
> **作者**：性能剖析 + 代码审计
> **目的**：明确"单次 input 流程"的具体调用链、各节点耗时、瓶颈、可优化点
> **结论**：单次 input 实际 **~14s**（稳定态），**不是 53.9s**；53.9s 是一次性外部 LLM 冷启动（SSL 握手+路由）。瓶颈是 **LLM#1 条件重入** + **隐藏的 voice_options LLM**。
> **v2.7+ 实验结论**：预取+1轮方案**有反噬效应**——让 LLM 主动探索更多 tool，LLM#1 调次数不降反升，**总耗时没改善**。已回滚到原 ReAct 行为。

---

## 0. TL;DR

| 指标 | 数值 |
|---|---|
| 首次冷启动（含 SSL/路由） | 53.9s（一次性，**外部 LLM 服务冷启动**） |
| 稳定单次 input 流程 | **10-14s** |
| LLM#1 (situation_assessment) | 2-7s × **1-2 次**（取决于 LLM 是否决定调 tool） |
| LLM#2 (narrative_fusion) | 3-4s × **1 次** |
| LLM#3 (voice_options 兜底) | ~3s × **1 次**（隐藏！） |
| 准备阶段（state_ref / system_prompt） | < 5ms |
| ToolNode（本地工具） | < 1ms |
| 真正可优化空间 | **2-4s**（减少 LLM#1 重入 + 合并 voice_options 到 LLM#2） |

## 3. 关键发现

### 3.1 LLM#1 经常调 2 次（ReAct 风格）

LangGraph 的"if tool_calls → ToolNode → back to situation_assessment"模式是 **ReAct 风格**——LLM 在第 1 轮决定调 tool，第 2 轮判断信息已足才放行。

**实测对比**：
- 冷启动第 1 次：tool_calls=2（调 2 个 tool），第 2 轮 tool_calls=0
- 热启动：tool_calls=0（一次到位，因为前一回合的工具结果已满足）

**优化方向**：
1. **缓存 tool 结果**（基于 state_ref hash）—— 同样状态下不重复调 tool
2. **prompt 改写**——告诉 LLM "在 system_prompt 阶段就预先 get_state()"，避免分多轮
3. **预测式并行**——把 get_state / check_rules / query_knowledge 改成"无条件预取"

### 3.2 voice_options 兜底 LLM 是隐形 N+1

[`web_server/routers/input.py:140-149`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/web_server/routers/input.py#L140-L149)：

```python
if not game.state.last_voice_options and last:
    try:
        extracted = merge_voice_options(None, last.get("narrative", ""))
        if extracted:
            # 命中本地正则提取，0 ms
            game.state.last_voice_options = extracted
        else:
            # LLM 驱动：再调一次 LLM 提取 ！~3s
            game.state.last_voice_options = _context_aware_voices(last.get("narrative", ""), game=game)
    except Exception as e:
        # 异常 fallback 也调 LLM
        game.state.last_voice_options = _context_aware_voices(last.get("narrative", ""), game=game)
```

**问题**：
1. 每次 input 流程**多调 1 次 LLM**（~3s）
2. 即使 `game.state.last_voice_options` 已被 LLM#2 填上，也会**重新覆盖**——因为 if 条件检查的是"是否为空"
3. **完全可以合并到 LLM#2**——让 narrative_fusion 的输出同时带 voice_options 字段

**代码位置**：[`input.py:221-300`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/web_server/routers/input.py#L221-L300) 的 `_context_aware_voices` + `_llm_generate_voices`

### 3.3 53.9s 的"冷启动"是外部 LLM 服务问题

LLM 第一次 ping 包含：
- TCP 握手
- TLS 握手
- HTTP/2 路由
- 服务端模型冷加载（如果是 serverless）

后续 ping 只 1.8s。所以 53.9s 是**一次性惩罚**，不影响 UX（用户不会在第 0 秒就 input）。

**优化方向**：
- 服务启动时**预热** LLM 连接（`_llm.invoke("ping")`）
- 玩家进入首页时就开始预热（不阻塞主线程）

### 3.4 system_prompt 每次重新构造

`_build_system_prompt()` 每次 0-1ms，但字符串拼接成本其实不低——平均 5KB，拼接 30 个 section。

**潜在优化**：把 system_prompt 缓存（key = era_id + identity + selected_identity），不变量就不重建。看 [`agent.py:1342-1350`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/dm_agent/agent.py#L1342-L1350)：

```python
# 已使用 kv_cache.SYSTEM_PROMPT_CACHE（@cached）
```

**但 cache 只对相同输入生效**——如果 state 变（时间、天气、关系），cache 命中率低。详见 `kv_cache.py`。

### 3.5 LLM#1 第 1 次 vs 第 2 次的时间差异

- 冷启动 LLM#1#1: 2.4s（调 tool）
- 冷启动 LLM#1#2: 5.0s（决策"够了"）

第 2 次反而**更慢**——因为 prompts 累积了 tool 结果，上下文更长。这是 LangGraph ReAct 的固有代价。

## 4. 优化建议（按 ROI 排序）

### 4.1 P0: 合并 voice_options 到 LLM#2（节省 3s）

**改动**：
- [`input.py:140-149`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/web_server/routers/input.py#L140-L149) 后处理**完全删除** voice_options 兜底
- 让 LLM#2 的输出 schema 增加 `voice_options` 字段（3-5 个 dict）
- 在 [`format_state.py`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/web_server/views/format_state.py) 序列化时透传

**节省**：3s（~22% 总体）
**风险**：低——LLM 一次输出多一个字段，prompt 简单调整
**工作量**：1-2 天

### 4.2 P1: 启动时 LLM 预热（消除 53.9s 冷启动）

**改动**：
- [`web_server/__init__.py:run()`](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/web_server/__init__.py) 启动时
- 异步调一次 `llm.invoke("hi")` 触发外部 LLM 路由

**节省**：用户首次 input 时的 50s
**风险**：低
**工作量**：0.5 天

### 4.3 P1: tool 结果按 state_ref hash 缓存（节省 5s）

**改动**：
- 在 LangGraph 节点前加一个 `cache_lookup_node`
- 缓存 key = sha1(state_ref)
- 缓存 value = tool_results
- 命中时跳过 LLM#1 tool calls

**节省**：1-2 次 LLM#1（最多 5s）
**风险**：中——cache 失效时反而会过时
**工作量**：2-3 天

### 4.4 P2: tool 调用条件化（让 LLM#1 一次到位）

**改动**：
- prompt 加规则："if player_input contains 'query X'，**直接**调 query_knowledge()"
- 减少 LLM 在第 1 轮需要做决策的复杂度

**节省**：可能 1-2s（如果成功）
**风险**：中——可能让 LLM 行为僵化
**工作量**：1 天

### 4.5 P3: 渐进式 LLM 流式输出

**改动**：
- LLM#2 改 streaming
- 前端 SSE/WebSocket 实时显示 narrative
- 用户感知 < 1s 就有第一个字

**节省**：感知 ~10s（实际 HTTP 不变，但用户看到第一字快）
**风险**：高——要改前后端协议
**工作量**：1 周

---

## 5. 不优化的边界

明确**不优化**：
- **LLM 本身速度**——外部服务决定，我们只能等
- **LangGraph 内部开销**——已经是 < 1ms
- **网络往返**——本地服务，最小化延迟

## 6. 推荐实施顺序

| 阶段 | 项 | 工作量 | 节省 | 累计 |
|---|---|---|---|---|
| **MVP** | P0 合并 voice_options | 1-2 天 | 3s | 3s |
| **MVP** | P1 启动预热 | 0.5 天 | 50s（首） | 50s 首 / 3s 后 |
| 增强 | P1 tool 结果缓存 | 2-3 天 | 5s | 8s |
| 增强 | P2 tool 条件化 | 1 天 | 1-2s | 9-10s |
| 重构 | P3 流式输出 | 1 周 | 感知 10s | 感知即时 |

**MVP（2-3 天）= 稳定态从 14s 降到 11s** + **首调 53s 降到 3s**（预热后）
