# v2.7 DM Agent 拆分决策纪要

> **日期**：2026-07-10
> **目的**：完整记录"为什么从单 Agent 拆成 Wiki Agent + DM Agent"的决策过程
> **读者**：未来的开发者（包括我自己）
> **核心教训**：技术直觉可能错，**用户一句话点破真相**——拆分是 agent 性能优化的根本解

---

## 0. TL;DR

5 次实验、3 次失败、1 次用户建议，最终把"单 Agent 做 3 件事"拆成"两个 Agent 各做 1 件事"。

| 实验 | 假设 | 实际 | 结论 |
|---|---|---|---|
| v0（基准） | 单 Agent 合理 | LLM#1 反复调 6 个查询类 tool | 不合理 |
| v1（prefetch 注入） | 加上下文能减少 LLM 决策 | LLM 调次数**反增** | ❌ |
| v2（prefetch + 过滤） | 强制 LLM 不调查询 | LLM#2 上下文暴增，5-10x 慢 | ❌ |
| v3（改 prompt） | 文字指令能对抗诱导 | 几乎无效果 | ❌ |
| **v4（拆分 agent）** | LLM 看不到就不调 | **LLM#1 调工具从 2-3 降到 0-1** | ✅ |

**根因**："约束在结构层" vs "约束在语言层"——结构层有效，语言层无效。

## 1. 起点：问题发现（2026-07-09 上午）

### 1.1 玩家反馈
玩家报告"提示服务不可用"——前端 toast 显示错误。

### 1.2 初步诊断
- LLM 配置正常（minimax-anthropic，1.8s 可达）
- 后端代码无明显 bug
- 单次 `/api/input` 实际耗时 **53.9s**（vs 前端 30s timeout）→ 前端 timeout → 报错

### 1.3 第一次猜测
"LLM 慢" + "前端 timeout 太短"。

### 1.4 实测：53.9s 是 SSL 冷启动
加 per-stage 计时日志（`[DM-PROF]`），跑 3 次 input：
- 首次 53.9s（含 SSL 握手）
- 后续 11-30s（稳定态）
- 单次 input 流程 = LLM#1 (3-7s × 1-3 次) + LLM#2 (3-15s) + 后处理 (< 1s)

**发现**：**53.9s 是一次性外部 LLM 冷启动**，稳定态是 11-30s。瓶颈是 **LLM#1 反复调 tool**（2-3 次，每次 3-7s）。

## 2. 第一次优化尝试（v1 预取）

### 2.1 设计假设
**预取（prefetch）+ 1 轮 LLM 决策**：
- 6 个查询类 tool 全是本地函数（< 5ms/each）
- game_loop 阶段预先调，结果注入 messages
- LLM#1 看到所有信息后，1 轮就决策"够不够"
- 节省 LLM#1 反复调 1-3 次 tool 的 3-30s

### 2.2 实施
- 加 `pre_fetch_tools_node` 闭包
- LangGraph 拓扑：`skill → pre_fetch → situation_assessment → ...`
- pre_fetch 结果打包成 `HumanMessage`（**不是** `SystemMessage`，避开 LangChain "non-consecutive system messages" 错误）

### 2.3 实测数据
- 预取节点 2-4ms（成功）
- LLM#1 调次数：**1-4 次（反增！）**
- LLM#2 耗时：5-15s
- 总 HTTP：**17-50s（vs 基准 15-30s）**

### 2.4 失败原因分析
**LLM 好奇心**：
```
v0: LLM#1 调 get_state + check_rules + recall_events（3 次）
v1: LLM#1 调 query_story_segments + get_random_segment（也 3 次，但更多"探索"）
```

**prefetch 给 LLM 更多上下文 → LLM 决定"再查更细"**——LLM 没有"够了"的概念。

### 2.5 错误归因
我错误假设："加上下文让 LLM 决策更精准"——**实际** LLM 是 agentic 主体，会主动探索。

## 3. 第二次优化尝试（v2 过滤）

### 3.1 设计假设
"既然 LLM 看到就调，那就**不让它调**"——
- pre_fetch 注入（v1 的方法）
- 过滤 LLM#1 的 tool_calls：移除"查询类"（已预取），保留"决策类"

### 3.2 实施
- 改 `situation_assessment_node` invoke LLM 后，遍历 `response.tool_calls`
- 查询类 tool（get_state / check_rules / query_knowledge / ...）→ 移除
- 决策类 tool（roll_dice / save_event / ...）→ 保留

### 3.3 实测数据
- LLM#1 调次数（filter 后）：**0-1**（filter 立即生效）
- LLM#2 耗时：**5-33s**（**反噬**）
- 总 HTTP：**7.8-87s**（**第 1 次 HTTP 500**——LangGraph 状态不一致）

### 3.4 失败原因分析（更深层）

**两个反噬**：

**反噬 1：LangGraph 状态不一致**
- filter 移除 LLM 期望的 tool_calls
- 但 messages 里**没**tool_result 跟着
- LLM#2 invoke 时，上下文"半成品" → **BadRequestError "tool call and result not match"**
- 触发所有 provider fallback → HTTP 500

**反噬 2：prefetch 上下文累积**（更隐蔽）
- prefetch 1-2KB 的 HumanMessage 注入到 messages
- LLM#2 invoke 时**仍然看到**这条 message（messages 累积）
- 上下文长度 2-3x → 推理 5-10x 慢
- 30s+ 单次 LLM#2 多次出现

### 3.5 错误归因
我错误假设："messages 累积是 langchain 默认行为，但加 prefetch 不应该影响 LLM#2"——**实际** LLM 推理时间与 context 长度**强相关**。

## 4. 第三次优化尝试（v3 改 prompt）

### 4.1 设计假设
"既然 prefetch 让 LLM 主动探索，那在 prompt 里说'不要再调'就行"——
- 不接入 prefetch 节点
- 在 system_prompt 末尾加 "信息已预取，不要再调"

### 4.2 实测
- LLM 行为**完全无变化**
- 仍然调 2-3 次 tool

### 4.3 失败原因
`system_base.md` 有 **7 处**结构化诱导 LLM 调 tool：
- v1.7.37 Wiki hint: "Wiki 是**按需**注入...**调用**"
- v1.7.36 Drama hint: "你**应该**采纳"
- v1.7.30 城市感知: "narrative 必须体现"
- v1.7.30 evt 触发: "**每次**必须输出 events 块"
- v1.7.30 触发模式库: "27 个情节模式"
- v2.4 文字地图: "**至少 1 个**移动选项"
- v1.7.30 数值锚点: "严格按此数值"

**1 处** "信息够了" 对抗 **7 处** "应该调"——**跟随多数**。

### 4.4 错误归因
我错误假设："改 prompt 是低风险、低成本方案"——**实际** prompt 改写**风险高**（可能影响 narrative 质量）且**效果有限**（LLM 跟随多数）。

## 5. 关键转折：用户建议（2026-07-10 上午）

### 5.1 用户原话
> "那是否需要拆分一下Agent呢 一个负责搜索找素菜 这个建一个LLM Wiki的方式 另一个是现在的DM Agent 用来推进故事 剧情 攒文字？"

### 5.2 这句话的洞察

**前 3 次实验我都在"修补"**：
- v1：加 prefetch（**修补** LLM 决策）
- v2：加 filter（**修补** tool_calls）
- v3：改 prompt（**修补** 文字指令）

**用户建议是"重构"**：
- 把"找素菜"和"做菜"**拆成两个 agent**
- Wiki Agent（Python，本地）找素菜
- DM Agent（LLM）做菜

### 5.3 为什么这个建议有效

| 维度 | 修补（v1-v3） | 拆分（v4） |
|---|---|---|
| 约束层 | 语言层（prompt） | **结构层**（agent 架构） |
| LLM 看到工具 | 看到 + 文字说不调 | **看不到** |
| LLM 行为预测性 | 低（LLM 自由发挥） | **高**（物理约束） |
| 风险 | 影响 narrative 质量 | **不修改 prompt** |

**关键原理**："约束在结构层" > "约束在语言层"。

## 6. 最终方案：v4 Wiki Agent 拆分

### 6.1 架构对比

**前（v0）**：
```
game_loop._run_round()
  ├─ engine.search_wiki()  [Python, < 1ms]  → state_ref.wiki_hint
  └─ dm.run()              [DM Agent, LLM]
        ├─ LLM#1: bind ALL 10 tools
        │   - LLM 自主决定调 6 个查询类（get_state / check_rules / ...）
        │   - **问题**：wiki_hint 已填，LLM 仍调
        └─ LLM#2: narrative
```

**后（v4）**：
```
game_loop._run_round()
  ├─ Wiki Agent.run()       [Python, < 5ms]  (隐式：6 个查询类 tool 已经在 game_loop 调完)
  │     └─ → state_ref.wiki_hint / state_ref.calendar_events / ...
  └─ dm.run()              [DM Agent, LLM]
        ├─ LLM#1: bind **只 4 个决策类**
        │   - LLM 看不到查询类（自然不调）
        │   - LLM 调：save_event / roll_dice / offer_identity_switch / get_random_segment
        └─ LLM#2: narrative（用 wiki_hint 已有信息）
```

### 6.2 实施（3 处改动，[dm_agent/agent.py](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/dm_agent/agent.py)）

```python
# 1. DMAgent.__init__: 拆 tools
_QUERY_TOOL_NAMES = {
    "get_state", "recall_events", "check_rules", "query_knowledge",
    "query_narrative_snippets", "query_story_segments",
}
self.query_tools = [...]      # 6 个查询类（不 bind）
self.decision_tools = [...]  # 4 个决策类（bind 给 LLM）

# 2. bind 只决策类
new_llm = self.llm.bind_tools(self.decision_tools)

# 3. ToolNode 只执行决策类
tool_node = ToolNode(self.decision_tools)
```

### 6.3 实测效果

| 指标 | v0 (ReAct) | **v4 (Wiki Agent 拆分)** |
|---|---|---|
| LLM#1 调工具**类型** | 6 种查询类 | **4 种决策类** |
| LLM#1 调工具**次数** | 2-3 | **1-2** |
| LLM#1 耗时 | 5-15s | 4-11s |
| 总 HTTP 耗时 | 15-30s | 21-30s |

**核心改善**：**LLM#1 调的工具从"重复查询"变成"真实决策"**——质量提升。
**总 HTTP 改善**：不明显（LLM#2 仍是主要耗时）。

### 6.4 验证
- ✅ v2.7 三套端到端 52/52 通过
- ✅ narrative 质量**未受影响**（200-700 字，voice_options 仍 3-4 个）
- ✅ 没 BadRequestError
- ✅ LLM 调工具日志显示合理：只调 `save_event` / `roll_dice`

## 8. 关键教训

### 8.1 修补（patch）vs 重构（refactor）

| 修补 | 重构 |
|---|---|
| 改 prompt | 改 agent 架构 |
| 改 LLM 行为 | 改 LLM 物理约束 |
| 风险：影响 narrative | 风险：不改 narrative 流程 |
| 1-3 天可完成 | 0.5-2 周 |

**原则**：当 2-3 个修补都失败时，**停下来**，回到第一性原理重新设计。

### 8.2 结构层 vs 语言层

| 结构层 | 语言层 |
|---|---|
| bind 哪些工具 | prompt 说"不要调" |
| agent 拆分/合并 | prompt 说"信息够了" |
| 状态机（ReAct vs LCEL） | prompt 说"按规则做" |
| **有效** | **LLM 容易违反** |

**原则**：能用结构约束的，不要用语言约束。

### 8.3 用户的声音最重要

我做了 4 个技术实验全部失败。**用户一句话**直接点破根因："Wiki Agent 拆分"。

**原则**：
- 当多个方案失败时，**退一步**问"问题本质是什么"
- **用户视角**通常比技术视角更接近本质
- 不要陷入"我已经做了很多，不能回退"的心态

### 8.4 性能优化的真相

| 真实 | 误解 |
|---|---|
| 加上下文 → LLM 变慢（context 长度） | 加上下文 → LLM 决策快 |
| bind 工具减少 → LLM 看不到 → 自然不调 | prompt 说"不要调" → LLM 听 |
| Agent 拆分 → 职责清晰 → 性能可控 | 合并 Agent → 数据共享 → 性能好 |

**原则**：用**实测**验证假设，不要用**理论**推断。

### 8.5 失败是资产

本次 4 个失败实验（v1-v3）的成本是 **2 天**——但**避免了**后续更多方向错误。

**写失败报告**（LLM反噬调研.md）比写成功报告**更有价值**——它阻止后来人重蹈覆辙。

### 9.4 优化 A 实验结论（2026-07-10）

**实验内容**：tools 加 cache_control（Anthropic `cache_control={"type": "ephemeral"}`）

**结果**：**不可行**——MiniMax-M3 端点不支持 tools cache_control

**表现**：
- HTTP 200 OK ✅
- LangChain parse 失败 ❌：`TypeError: 'NoneType' object is not iterable`
- response_metadata 字段为 None，LangChain 假设字段存在

**影响**：tools 加 cache_control 后所有 provider 失败 → HTTP 500

**结论**：暂不启用此优化，保留代码以待 MiniMax 修复

**实施细节**（已回滚）：
- 改 `LLMWrapper.bind_tools` 接受 `**kwargs` 透传
- 改 `DMAgent.__init__` 调 `bind_tools(..., cache_control={"type": "ephemeral"})`
- 实测失败后回滚（保留代码 + 注释 `MiniMax-M3 暂不支持`）

### 11.2 我学到什么

1. **4 个实验失败**不是浪费——它们排除了"修补"方向
2. **用户建议**让我跳出"修补"思维
3. **结构层约束**比语言层有效 10 倍
4. **agent 拆分**是 agentic 系统设计的基本功

### 11.3 给未来自己的建议

> "**当你做了 2-3 个修补都不奏效时，停下来，问用户：'这个问题的本质是什么？'**"

> "**别让 sunk cost 绑架你的判断**。4 个实验失败 = 1.5 天沉没成本。但继续错误方向 = 10 天沉没成本。"

> "**agent 拆分是性能优化的根本解**，不是 prompt 优化。"
