# v2.7+ Wiki Agent 拆分设计

> **日期**：2026-07-10
> **作者**：用户建议 + 实测验证
> **目的**：把"找素菜"和"做菜"分开——Wiki Agent（Python，< 5ms）查资料，DM Agent（LLM）只负责叙事
> **结论**：✅ 真正解决了"LLM 反复调查询类 tool"问题。LLM#1 调工具数从 2-3 降到 0-1。

---

## 0. TL;DR

| 状态 | LLM#1 调的工具 | 调次数 | 调工具内容 | HTTP 耗时 |
|---|---|---|---|---|
| v0 (ReAct, 全 bind) | 2-3 个 | 2-3 | get_state, check_rules, query_knowledge, query_narrative_snippets, ... | 15-30s |
| **v3 (Wiki Agent 拆分)** | **0-1 个** | **1-2** | **只调 save_event / roll_dice**（决策类） | 21-30s |

**核心变化**：DM Agent LLM 只 bind 4 个**决策类** tool（save_event / roll_dice / offer_identity_switch / get_random_segment）。**6 个查询类 tool**（get_state / check_rules / query_knowledge / ...）已被 game_loop 阶段的 Python Wiki Agent 预取，结果注入 `state_ref.wiki_hint` 等字段。LLM 看不到查询类工具，**自然不调**。

## 1. 拆分前的问题（v0 ReAct）

### 1.1 架构现状

```
game_loop._run_round(player_input)
  ├─ engine.search_wiki()           [Python, < 1ms]  → state_ref.wiki_hint
  ├─ engine.check_calendar_events() [Python, < 1ms] → state_ref.calendar_events
  ├─ set_drama_hint_for_dm()        [Python, < 1ms] → state_ref.drama_hint
  └─ dm.run()                                       [DM Agent, LLM]
        ├─ LLM#1 (situation_assessment)
        │   - bind: ALL 10 tools
        │   - LLM 自主决定调 get_state / check_rules / query_knowledge / ...
        │   - **问题**：wiki_hint 已经填了，LLM 还是调
        └─ LLM#2 (narrative_fusion)
            - bind: ALL 10 tools
            - 写叙事
```

### 1.2 4 个失败实验

| 实验 | 方案 | 失败原因 |
|---|---|---|
| v1 | prefetch 注入 HumanMessage | LLM 好奇心，prefetch 让 LLM 调更多 |
| v2 | prefetch + tool_calls 过滤 | LLM#2 上下文暴增，5-10x 慢 |
| v3 | 只改 system_prompt 加"足够"指令 | 文字对抗不了 7 处结构化诱导 |

### 1.3 真正的原因

LLM 行为学：
- 给 LLM 工具描述，LLM **会**主动调（"工具的存在就是用来用的"）
- 给 LLM 更多上下文，LLM **会**想"再查更细"
- 文字指令**对抗不了**结构化诱导

**根本解法**：**别让 LLM 看到查询类工具**。

## 2. 拆分后（v3 Wiki Agent）

### 2.1 新架构

```
game_loop._run_round(player_input)
  ├─ Wiki Agent.run()                                [Python, < 5ms]
  │     ├─ get_state()                       [本地]
  │     ├─ recall_events()                   [本地]
  │     ├─ check_rules()                     [本地]
  │     ├─ query_knowledge()                 [本地]
  │     ├─ query_narrative_snippets()        [本地]
  │     └─ query_story_segments()            [本地]
  │     → 输出 wiki_hint / state_ref 各字段
  │
  └─ DM Agent.run()                                  [DM Agent, LLM]
        ├─ LLM#1 (situation_assessment)
        │   - bind: **只 4 个决策类** tool
        │   - LLM 看不到查询类（自然不调）
        │   - LLM 只调：save_event / roll_dice / offer_identity_switch / get_random_segment
        └─ LLM#2 (narrative_fusion)
            - bind: **只 4 个决策类** tool
            - 写叙事（用 wiki_hint / state_ref 已经有的信息）
```

### 2.2 关键改动

| 改动 | 文件 | 内容 |
|---|---|---|
| 拆分 tools | `dm_agent/agent.py` DMAgent.__init__ | 10 tool 拆成 query_tools (6) + decision_tools (4) |
| 只 bind 决策类 | `dm_agent/agent.py` __init__ + _build_graph | `llm.bind_tools(decision_tools)` |
| ToolNode 只执行决策类 | `dm_agent/agent.py` _build_graph | `ToolNode(decision_tools)` |

### 2.3 4 个决策类 tool

| Tool | 何时调 | 保留原因 |
|---|---|---|
| `save_event` | LLM 写完 narrative | v1.7.30 残留要求（**未来可改 game_loop 自动写**） |
| `roll_dice` | LLM 决定需要检定 | 决策类 |
| `offer_identity_switch` | LLM 想给玩家身份切换 | 决策类 |
| `get_random_segment` | LLM 想随机取一段叙事片段 | 决策类（不过 LLM 也可以从 wiki_hint 拿） |

### 2.4 Wiki Agent 复用 game_loop 已有的预取

**重要**：v1.7.37 **已经**在 game_loop 阶段**预取 wiki_fragments**（[game_loop.py:358-368](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/game_loop.py#L358-L368)）——`engine.search_wiki()` Python 直接调，结果注入 `state_ref.wiki_hint`。

**v2.7+ Wiki Agent 拆分**是**扩展**这个思路：
- 之前只预取 1 个 tool（query_knowledge）
- 现在 DM Agent LLM **完全看不到** 6 个查询类 tool，**所有查询结果来自 game_loop 预取**

### 2.5 实测数据

| 指标 | v0 (ReAct) | v3 (Wiki Agent 拆分) |
|---|---|---|
| LLM#1 调的工具 | 2-3 个 | **0-1 个** |
| LLM#1 调次数 | 2-3 | 1-2 |
| LLM#1 调工具类型 | get_state, check_rules, query_knowledge, ... | **save_event, roll_dice**（仅决策类） |
| LLM#1 耗时 | 5-15s | 4-11s |
| LLM#2 耗时 | 5-10s | 5-15s |
| **总 HTTP 耗时** | **15-30s** | **21-30s** |

**改善**：
- ✅ LLM#1 调工具**完全合理化**（不再调查询类）
- ✅ LLM#1 工具调**次数减半**
- ⚠ 总 HTTP 耗时改善不显著（因为 LLM#2 仍是主要耗时）

**为什么总耗时没大幅改善**：
- 之前 v0 LLM#1 调查询类 tool 实际是**反噬**（调 2-3 次但都是本地函数，0ms 执行；3-15s 是 LLM 决策 + 回复）
- 现在 v3 LLM#1 调决策类 tool 实际**真实有用**（save_event / roll_dice 需要 LLM 决策；1-2 次就够）
- LLM#1 决策次数没真正减少（仍在 situation_assessment 来回），只是工具类型变了

**真要省时间**：
- LLM#1 单次调 4-11s（决策类 1-2 次）≈ LLM#1 单次调 5-15s（查询类 2-3 次）
- **减少的是 LLM#1 工具多样性**（不再去"调 6 个查询"），**不是 LLM#1 调次数**

## 3. 风险评估

### 3.1 narrative 质量

**理论风险**：LLM 看不到查询类 tool，可能不知道有这些信息可用。

**实测**：
- ✅ 仍能输出 200-700 字 narrative
- ✅ voice_options 仍 3-4 个
- ✅ query_knowledge 仍通过 wiki_hint 注入（v1.7.37 已有机制）
- ✅ narrative 内容质量**没明显变化**（player 体验上）

### 3.2 决策类 tool 调用

LLM 仍能调 `save_event` / `roll_dice` / `offer_identity_switch` / `get_random_segment`——这些是真正的决策。

**实测**：仍能调 `save_event`（v1.7.30 残留要求）、`roll_dice`（自动检定）。

### 3.3 异常处理

如果 LLM 想调查询类 tool（bind 不上）—— LLM 会**理解**为"没这工具"，**直接生成 narrative**。已经实测过：**没有 BadRequestError**。

### 3.4 v1.7.30 残留 save_event 要求

prompt v1.7.30 段 377-419 说"每次 narrative 后必须输出 events 块"——**这与 v1.7.32 架构变更冲突**（v1.7.32 说"LLM 不需要输出 events"）。

**结果**：LLM 仍调 `save_event` 工具（v1.7.30 残留诱导）。

**真解法**：改 prompt 删除 v1.7.30 events 块要求（高风险，需要 A/B 测试）。

**当前状态**：保留 save_event 工具（兼容旧 prompt），LMM 调它 1-2 次/回合，**总耗时 1-3s**，可接受。

## 4. 未来可能优化

### 4.1 P0：把 LLM#1 单次直接进 narrative（消除 situation_assessment 循环）

**方案**：v3 基础上，situation_assessment 节点**只 bind 决策类**（已经做了），但**进一步**强制 LLM 单次返回（不要 call_tools）——让 should_continue 在第 1 轮就强制 narrative_fusion。

**风险**：LLM 必须在第 1 轮就生成 narrative——没机会调 save_event / roll_dice。**质量可能下降**。

**缓解**：把 save_event 改 game_loop 自动调（不需要 LLM 调）；roll_dice 也 game_loop 自动检定。

### 4.2 P1：删 save_event 工具，改 game_loop 自动写事件

**v1.7.32 架构**已经说"游戏引擎处理所有结构化数据"——save_event 应该是 game_loop 责任，**不是 LLM**。

**方案**：
- 删 save_event tool
- game_loop `_run_round` 末尾自动从 narrative 提取事件，调用 memory.save_event

**节省**：消除 LLM 调 save_event 1-2 次（2-6s）

**风险**：event_parser 提取可能不如 LLM 灵活。需要测试。

### 4.3 P2：删 get_random_segment 工具，改 game_loop 自动注入

**v1.7.37 Wiki 检索**已经预取 query_story_segments——get_random_segment 是冗余的。

**方案**：
- 删 get_random_segment
- game_loop 阶段随机选 1 段注入 wiki_hint

**节省**：消除 LLM 调 get_random_segment 0-1 次（0-3s）

### 4.4 P3：换 LCEL agent（彻底拆分）

**长期**：放弃 LangGraph ReAct，换 LangChain Expression Language (LCEL) 的 `RunnablePassthrough`：

```python
# game_loop 阶段（Python Wiki Agent）
wiki_context = WikiAgent.run(player_input, state)  # < 5ms

# DM Agent（一次 LLM call）
narrative = dm_llm.invoke({
    "system": system_prompt,
    "wiki_context": wiki_context,
    "player_input": player_input,
})  # < 10s
```

**总耗时**：< 15s（对比现在 21-30s）

**风险**：1-2 周工作量，narrative 质量可能变化。

## 6. 关键教训

### 6.1 用户建议最有效

我做了 4 个实验（prefetch 接入、prefetch + filter、system_prompt 改写）**全部失败**。**用户一句话点破**：拆分 Wiki Agent。

**启示**：当技术方案不奏效时，**回到第一性原理**重新设计架构比修补 prompt 有效。

### 6.2 "看不到"比"看到+指令"更有效

- 改 prompt 让 LLM "不要再调查询类" → 失败
- 让 LLM 物理上**看不到**查询类 → 成功

**原则**：**约束应在结构层，不在语言层**。

### 6.3 拆分是 agentic 设计的正确方向

当前 DM agent 做了"找素菜 + 做菜 + 上菜"3 件事——LLM 既要查知识又要叙事。**职责过多**。

正确设计：Wiki Agent 找素菜（Python 快，5ms），DM Agent 做菜（LLM 慢，10-15s 但只做"做菜"）。

### 6.4 不能 100% 控制 LLM 行为

即使 bind 只决策类，LLM **仍** 调 save_event（v1.7.30 残留 prompt 诱导）。**要彻底干净需要改 prompt**。

**短期接受**（当前）：LLM 调 save_event 1-2 次/回合（1-3s），可接受。
**长期解法**：删 v1.7.30 残留 + 改 game_loop 自动 save_event。
