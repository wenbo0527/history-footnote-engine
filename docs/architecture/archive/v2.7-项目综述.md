## 0. 一句话总结

**历史注脚**（History Footnote）是一个 AI 驱动的高自由度历史 RPG 体验引擎。玩家扮演万历十五年盛泽镇织户，5-15 回合做完一个完整"过日子"循环（含织绸、卖绸、出行、社交、事件触发、命运牌局、结算），产生可用于社交分享的"角色卡"。

**核心技术**：LangGraph ReAct DM Agent + MiniMax-M3 LLM + 8 套 AI DM 技能 + 多层缓存。

### 2.2 单次 `/api/input` 完整调用链（v2.7+ Wiki Agent 拆分后）

```
POST /api/input { session_id, input: "我检查织机" }
  ↓
input.py handle_POST_input
  ├─ AP/HP/状态预处理
  ↓
game_loop._run_round()
  ├─ engine.search_wiki()  [Python, 0-50ms]   ← Wiki Agent（隐式）
  ├─ check_rules / events (引擎计算，非 LLM)
  ├─ LLM_THROTTLE.acquire (semaphore max=3)
  ├─ LLM_THROTTLE.invoke (timeout 90s, retry 3)
  ↓
DMAgent.run(player_input)
  ├─ 准备 state_ref（玩家状态/位置/资源/已触发事件）
  ├─ _build_system_prompt (700+ 行 → 5KB cache)
  ├─ graph.invoke()
  │   ├─ skill_orchestration_node
  │   ├─ situation_assessment_node → LLM#1 (4-12s)
  │   │   ├─ bind 4 决策类 tool (save_event/roll_dice/offer_identity_switch/get_random_segment)
  │   │   ├─ 6 个查询类 tool 已**物理不可见**（v2.7+ 拆分后）
  │   │   ├─ LLM 自由决策，调决策类 tool 0-1 次
  │   ├─ should_continue → ToolNode（如有 tool_calls）
  │   ├─ narrative_fusion_node → LLM#2 (4-15s)
  │   ├─ extract_narrative_node
  └─ state.apply_changes
  ↓
format_state.py 序列化（200-700 字 narrative + 3-4 voice_options）
  ↓
handler._json(200, ...) ← HTTP 响应
  ─────────
  总耗时：稳定态 16-41s（中位 32.4s）
```

### 2.3 关键模块依赖图

```
web_server/routers/input.py
    └─→ game_loop._run_round()
          ├─→ engine.search_wiki (Python, knowledge_base)
          ├─→ DMAgent.run()
          │     ├─→ 10 个 tool（get_state/recall_events/check_rules/query_knowledge/...）
          │     │     ├─→ game_state
          │     │     ├─→ game_memory
          │     │     ├─→ rule_engine
          │     │     └─→ knowledge_base
          │     └─→ llm_with_tools → llm_wrapper.invoke()
          │           └─→ provider.invoke() (MiniMax-M3 / DeepSeek / mock)
          ├─→ format_state.py
          ├─→ save_state (异步落盘)
          └─→ voice_options 注入（context-aware fallback）
```

## 5. 教训（来自 v2.7 1.5 天密集迭代）

### 5.1 技术层教训

1. **agent 性能优化 ≠ prompt 优化**：架构问题（ReAct 循环、messages 累积）不是 prompt 措辞能解决
2. **结构层 > 语言层**：bind 哪些工具是结构层（有效），prompt 说"不要调"是语言层（LLM 容易违反）
3. **实测 > 理论**：我之前纸上分析 13s/次节省，实际反噬
4. **可回滚比快速优化重要**：所有改动默认关闭，出问题可回滚
5. **失败是资产**：4 个失败实验（v1-v3）= 1.5 天成本，但避免了后续更多方向错误

### 5.2 工程层教训

1. **用户声音最重要**：4 个技术实验失败 → 用户一句话点破根因（Wiki Agent 拆分）
2. **性能剖析先于性能优化**：`[DM-PROF]` 日志保留下来，未来继续用
3. **3 个模型特定优化 vs 3 个通用优化**：换模型时需重新评估 6 项中的 3 项

# 🆕 v2.7+ 启动 LLM 预热（消除首次 /api/input 的 53.9s SSL 冷启动）
def _warmup_llm() -> None:
    try:
        from history_footnote.llm_providers import make_llm_for_purpose
        from history_footnote.llm_wrapper import LLMWrapper
        logger.info("[v2.7+ warmup] 启动 LLM 预热")
        wrapped = LLMWrapper(
            primary_provider="minimax-anthropic",
            timeout=60.0,
            retry_on_same=1,
        )
        t0 = time.time()
        wrapped.invoke(
            [{"role": "user", "content": "hi"}],
            timeout=60.0,
        )
        dt = (time.time() - t0) * 1000
        logger.info(f"[v2.7+ warmup] ✅ 预热完成（{dt:.0f}ms）")
    except Exception as e:
        logger.warning(f"[v2.7+ warmup] ⚠️ 预热失败（不影响启动）: {e}")

threading.Thread(target=_warmup_llm, daemon=True, name="llm-warmup").start()
