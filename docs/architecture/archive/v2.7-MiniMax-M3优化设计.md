## 2. 优化 A：tools 加 cache_control（5 分钟可做）

> **⚠️ 本优化是 MiniMax-M3 + Anthropic 兼容 API 特有**——`cache_control` 是 Anthropic API 字段。换 OpenAI 协议不适用。

### 2.1 现状
- `system_prompt` 用了 cache_control（`[kv_cache.py:128](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/kv_cache.py#L128)`）
- `tools`（bind_tools 传入）**没用** cache_control
- `state_ref`（每回合变）**不能用** cache_control

### 2.2 问题
每次 LLM#1 invoke 重新传 **4 个 tool schema**（约 2-3K tokens）—— **不缓存**。

50 回合后：
- cache miss 50 次 × 2-3K = 125K tokens 浪费
- 成本 $0.04（minimax 估算）—— 实际可能更高

### 2.3 修复（LangChain 标准做法）

LangChain 的 `bind_tools` 不直接支持 `cache_control`。需要：

```python
# 当前
llm_with_tools = self.llm.bind_tools(self.decision_tools)

# 优化后（手动构造）
from langchain_anthropic import ChatAnthropic

# ChatAnthropic 支持 .with_config 给 tools 加 cache_control
llm_with_tools = self.llm.bind_tools(self.decision_tools)
# 然后在 .invoke() 时改 raw payload
```

**实际方案**：用 `ChatAnthropic` 的 `extra_body` 参数：

```python
# llm_providers.py
def make_anthropic_llm(api_key, base_url, model, **kwargs):
    from langchain_anthropic import ChatAnthropic
    return ChatAnthropic(
        model=model,
        api_key=api_key,
        base_url=base_url,
        # 🆕 v2.7+ 启用 cache_control breakpoint（最多 4 个）
        # 1. system
        # 2. tools (这里)
        # 3. messages last
        extra_body={
            "tools": [
                {**t.dict(), "cache_control": {"type": "ephemeral"}}
                for t in decision_tools
            ]
        }
    )
```

### 2.4 节省估算

| 项 | 当前 | 优化后 |
|---|---|---|
| tools schema tokens | 2-3K/次 | cache read (1/3 价格) |
| 50 回合节省 | 0 | 100K-150K tokens × $0.06/M = **$0.01-0.02/50 回合** |

节省**少**（5-15% 成本），但**零风险**——纯加 cache 标记。

## 3. 优化 B：system + state_ref 拆 2 cache_control（节省 1-2s/次）

> **⚠️ 本优化是 MiniMax-M3 + Anthropic 兼容 API 特有**——依赖 Anthropic 的多 cache_control breakpoints 机制。

### 3.1 现状
`state_ref` 每回合变 → 整个 message 不能缓存。

### 3.2 优化思路
Anthropic API 支持**多 cache_control breakpoints**（最多 4 个）：

```
[system_prompt (cache)] [state_ref (no cache)] [messages (cache last msg)]
```

但当前 `[__STATE_REF__]` 是塞在 system_message 后面的（同一个 SystemMessage），**不能单独 cache**。

### 3.3 改造方案

把 `state_ref` 从 SystemMessage 拆到**单独的 HumanMessage**：

```python
# 旧
system_message = SystemMessage(content=[{"type": "text", "text": system_prompt, "cache_control": cache_control}])
state_ref_msg = SystemMessage(content=f"\n\n[__STATE_REF__]\n{_json.dumps(state_ref, ensure_ascii=False)}")
initial_state["messages"] = [system_message, state_ref_msg, HumanMessage(content=player_input)]

# 新
system_message = SystemMessage(content=[
    {"type": "text", "text": system_prompt, "cache_control": cache_control}  # ⬅ cache
])
state_ref_msg = HumanMessage(content=f"[__STATE_REF__]\n{_json.dumps(state_ref, ensure_ascii=False)}")
# 同样的拆分：state_ref 不进 cache（每回合变）
initial_state["messages"] = [system_message, state_ref_msg, HumanMessage(content=player_input)]
```

**为什么节省 1-2s**：state_ref 现在被作为 system 的一部分传 → 不能被独立 cache → cache 失效时 system_prompt 也要重传（5-8K tokens × 50 回合 = 浪费）。拆开后，**system_prompt 持续缓存**。

### 3.4 风险
LLM 看到 HumanMessage vs SystemMessage 行为可能不同——**实测验证**。

### 3.5 节省
**1-2s/次**（system_prompt 持续 cache，不被 state_ref 污染）

## 4. 优化 C：LLM#2 用 structured output（5-10s/次）

> **✅ 本优化适用于所有支持 tool_use 的 LLM**（包括 MiniMax-M3、Anthropic Claude、OpenAI GPT-4 等），但具体实施因模型而异（MiniMax-M3 用 Anthropic 协议）。

### 4.1 现状
LLM#2 输出**自由 JSON**（[system_base.md:113](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/dm/prompts/system_base.md#L113) 说"输出 JSON 含 narrative/voice_options 等 8 字段"），但 v1.7.32 段又说"不要输出 events/state_changes"——**冲突**。

后处理 `sanitize()` 容错：
```python
# narrative_sanitizer.py:367
parsed = json.loads(json_text)
if isinstance(parsed, dict) and "narrative" in parsed:
    return strip_skill_metadata(parsed["narrative"])
```

**LLM 自由发挥**：
- 经常输出 JSON 但字段错（narrative_blocks 写错、voice_options 字段名不对）
- 经常输出 `<narrative>` 标签 XML（v1.7.32 段说"输出 narrative 标签"）
- 经常输出 markdown 段落（不按 JSON）

**LLM 反复试错** → 推理慢 5-10s。

### 4.2 优化：tool schema 强制结构化输出

Anthropic API 的 `tool_choice={"type": "tool", "name": "submit_narrative"}` 强制 LLM 调特定 tool：

```python
# 伪代码
llm_with_narrative_tool = llm.bind_tools([
    {
        "name": "submit_narrative",
        "description": "提交最终叙事",
        "input_schema": {
            "type": "object",
            "properties": {
                "narrative": {"type": "string", "description": "300-500字叙事"},
                "narrative_blocks": {"type": "array", "items": {...}},  # 可选
                "voice_options": {"type": "array", "items": {...}},
                "is_action": {"type": "boolean"},
                "time_cost": {"type": "integer"},
                "intent_type": {"type": "string", "enum": ["action", "inquire", "describe", "voice"]},
            },
            "required": ["narrative", "voice_options", "is_action", "time_cost"]
        }
    }
], tool_choice={"type": "tool", "name": "submit_narrative"})
```

**LLM 调用**：
```python
response = llm_with_narrative_tool.invoke(messages)
# response.tool_calls[0]["args"] 已经是结构化 dict，无需解析
narrative_data = response.tool_calls[0]["args"]
```

### 4.3 优势

| 维度 | 旧（JSON 自由） | 新（tool schema） |
|---|---|---|
| 字段保证 | ❌ LLM 自由发挥 | ✅ schema 强制 |
| 字段类型 | ❌ 经常 string/int 错 | ✅ 类型检查 |
| 解析 | ❌ json.loads + 容错 | ✅ 直接 dict 访问 |
| 速度 | 慢（LLM 思考字段格式） | 快（schema 引导） |
| 鲁棒性 | ❌ 解析失败 50% | ✅ 几乎 0% 失败 |

### 4.4 节省
- **5-10s/次**（LLM#2 不再思考字段格式）
- **容错代码可删**（[narrative_sanitizer.py:367](file:///Users/mac/Documents/trae_projects/history_footnote/src/history_footnote/narrative_sanitizer.py#L367) 那一段）

### 4.5 风险
- Anthropic tool_use 严格模式可能让 LLM 输出变得**机械**（v1.7.32 说"LLM 自由发挥"）
- 需要 A/B 测试 narrative 质量

### 4.6 实施要点

```python
# dm_agent/agent.py narrative_fusion_node
def narrative_fusion_node(state: DMState) -> dict:
    # 🆕 v2.7+ 用 submit_narrative tool 强制结构化
    response = llm_with_narrative_tool.invoke(state["messages"])
    # tool_choice 强制 LLM 必调这个 tool
    # response.tool_calls[0]["args"] = {"narrative": "...", "voice_options": [...], ...}
    return {
        "messages": [response],
        "narrative": response.tool_calls[0]["args"]["narrative"],
        "voice_options": response.tool_calls[0]["args"]["voice_options"],
    }
```

## 5. 优化 D：并发调 6 个查询类 tool（1-2s/次）

> **✅ 本优化是模型无关的**——任何 LLM 都能受益。

### 5.1 现状
`pre_fetch_query_tools()` 在 `run()` 里**串行**调 6 个 tool（虽然都是本地函数，< 5ms 每个）：
- get_state → recall_events → check_rules → query_knowledge → query_narrative_snippets → query_story_segments
- 总耗时 2-4ms（实测）

**2-4ms 不值得并发**——**没有节省**。

### 5.2 但有 LLM 调用
**实际 LLM 端调 6 个 tool 不存在**（v4 拆分后，LLM 不调查询类）。**但 game_loop 阶段有 LLM 调用**（`engine.search_wiki`）—— 那个本身 ~1s。

### 5.3 真正可并发
**game_loop._run_round 阶段**：
- `engine.search_wiki` (LLM, ~1-3s)
- `engine.check_calendar_events` (LLM, ~1-3s)
- `engine.search_narrative_snippets` (LLM, ~1-3s)
- ...可能 3-5 个 LLM 调用

**并行**这 3-5 个 LLM 调用 → **3-5s → 1s**（节省 2-4s）

### 5.4 风险
- 并发 LLM 调用可能限流（已用 35/600s 限流）
- context 长度平方增长

### 5.5 节省
**2-4s/次**（并发 game_loop LLM 预取）

## 6. 优化 E：system_prompt 精简（0.5-1s/次）

> **✅ 本优化是模型无关的**——任何 LLM 都受益（短 prompt → 快 attention）。

### 6.1 现状
`system_base.md` **700+ 行**——多数场景不需要。

### 6.2 精简方向
- ❌ 删除 v1.7.30 events 块要求（v1.7.32 冲突段）— 节省 50 行
- ❌ 删除 v1.7.36 Drama hint（LLM 不调 tool，hint 失效）— 节省 30 行
- ❌ 删除 v1.7.30 城市感知（state_ref 已注入）— 节省 20 行
- ❌ 合并重复段落 — 节省 20 行

### 6.3 节省
- system_prompt tokens：~10K → ~7K（-30%）
- 单次 input 成本 -30%
- 单次 LLM 推理时间 **0.5-1s/次**（prompt 短 → attention 快）

### 6.4 风险
- 删多了影响 narrative 质量
- 需要 A/B 测试

### 6.5 建议
**先做 A/B 测试**——保留原版做对照组，新版做实验组，比较 narrative 长度/质量/玩家评分。

## 7. 优化 F：Extended Thinking（实验性）

> **⚠️ 本优化是 MiniMax-M3 + Anthropic 兼容 API 特有**——`thinking` 是 Anthropic API 参数。换 OpenAI 协议不适用（OpenAI 用 o1 / o3 内置推理模型）。

### 7.1 是什么
Anthropic API 的 `thinking` 参数（2024 年底 release）—— LLM 在回复前先内部"思考"，输出 `thinking` 块。

### 7.2 优势
- 复杂决策（DM narrative）更准
- 玩家感知"DM 思考更深"

### 7.3 劣势
- 增加 20-30% tokens（thinking 块也收费）
- 增加 1-3s 思考时间

### 7.4 评估
**不建议默认开**——成本不划算，DM narrative 不需要"深度思考"（已经有 wiki_hint + state_ref 提供足够信息）。

**未来可能**：复杂剧情决策（如 boss 战、特定事件触发）临时启用。
