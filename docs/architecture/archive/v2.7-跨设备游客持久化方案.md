# v2.7+ 跨设备游客持久化方案（设计文档）

> 目标：让游客在换浏览器/换设备时仍能找回自己的存档，同时不破坏现有账户体系的安全模型。

---

## 0. 现状摘要

| 维度 | 当前 |
|---|---|
| 游客 ID 来源 | 前端 localStorage `hfe_account_id`（仅本机） |
| 存档落盘 | `saves/wanli1587_<ts>/meta.json + slot*.json`，meta 含 `account_id` 字段 |
| 后端账号记录 | `saves/accounts/accounts.json`，游客 `role=guest`、`account_id=guest_<8hex>` |
| 注册迁移 | `POST /api/account/register` 接 `migrate_from_guest_id` 改写 meta.json owner |
| 跨设备 | ❌ 全部丢失（游客 ID 只在 localStorage） |
| 已有 cookie 体系 | ❌ 完全没用上 cookie（Handler 直接 BaseHTTPRequestHandler，未 Set-Cookie） |
| 已有会话/鉴权 | 管理员走 `_check_admin_token`，其他接口基本裸奔（按 IP 限流） |

**核心矛盾**：存档已经在服务端（`saves/`），但"谁拥有"的身份只存在浏览器本地——所以换浏览器就找不到。

---

## 1. 四种持久化方案对比

### 方案 A：HttpOnly 签名 Cookie

**原理**：游客第一次进首页时服务端 `Set-Cookie: hfe_guest=<signed_uuid>; HttpOnly; Secure; SameSite=Lax; Max-Age=...`，后续所有 API 从 cookie 取 ID。

| 维度 | 评价 |
|---|---|
| 跨设备 | ✅ 默认自动跟随（同浏览器同账号登录就带走） |
| 换设备 | ❌ 仍需账号登录态才能跨设备同步——只解决"游客换浏览器"问题 |
| 安全性 | ✅ HttpOnly 防 XSS 偷；签名防伪造；Secure 限制 HTTPS |
| 前端改造 | 极小：游客进入时不再需要"先调一次 /api/menu 拿 ID"，直接读 cookie |
| 后端改造 | 中：增加 cookie 签发/校验中间件、SecretKey 配置 |
| 隐私 | 服务端知道"这个浏览器对应这个游客 ID"，不收集 IP/UA |
| 缺点 | 清浏览器 cookie 即丢；隐私模式可能拒存 cookie；SameSite 跨站会丢 |

### 方案 B：恢复码（Recovery Code）

**原理**：游客进入时生成 `HF-RECOVER-XXXX-XXXX-XXXX` 12 字符恢复码，绑定 account_id。前端展示一次（"请抄下来"），用户可手动在新设备输入恢复码继续。

| 维度 | 评价 |
|---|---|
| 跨设备 | ✅ 一次性手动跨设备 |
| 换设备 | ✅ 强：靠"用户保管的密钥"，不依赖任何服务端状态机 |
| 安全性 | ✅ 12 字符 ~ 10^14 空间（62^12 ≈ 3×10^21，但限速 5/min 防爆破） |
| 前端改造 | 大：需要 UI 展示恢复码 + 输入框 + 恢复流程 |
| 后端改造 | 中：增加 `recovery_codes.json` 存储 + `POST /api/account/recover` |
| 隐私 | ⭐ 最强：服务端不知道哪个 ID 属于哪个真实人——除非用户主动恢复 |
| 缺点 | 用户必须主动抄码（容易丢）；UI 占用空间 |

### 方案 C：Cookie + 恢复码 双轨

| 维度 | 评价 |
|---|---|
| 跨设备 | ✅ Cookie 自动 |
| 换设备/清 cookie | ✅ 恢复码兜底 |
| 安全性 | ✅✅ 双重保险 |
| 前端改造 | 大：两套 UI |
| 后端改造 | 大：两套体系 |
| 隐私 | 中：默认服务端知道（cookie 路径），恢复码可主动脱钩 |
| 缺点 | 实现成本最高；对当前 v2.7 beta 阶段过度设计 |

### 方案 D：纯服务端按 IP+UA 绑定（不推荐）

服务端用 `client_address + User-Agent + Accept-Language` 哈希当 ID。

| 维度 | 评价 |
|---|---|
| 跨设备 | ❌ 换设备天然跨 |
| 换浏览器（同 IP） | ⚠️ 大概率丢（UA 变） |
| 安全性 | ⚠️ 网吧/公司共享 IP 互踩；移动网络 IP 漂移 |
| 隐私 | ❌ 服务端长期留存 IP+UA 哈希，GDPR/个保法风险 |
| 缺点 | **强烈不推荐**——既不安全也不可靠 |

## 2. 存储位置对比

| 方案 | 路径 | 优点 | 缺点 |
|---|---|---|---|
| 复用 `saves/wanli1587_<ts>/` | 不动 | 零迁移；与登录用户存档共享 SaveManager；备份/统计一致 | 区分"游客存档 vs 登录用户存档"靠 meta.json account_id（已实现） |
| 新建 `saves/guests/<guest_id>/<session_ts>/` | 物理隔离 | 一眼能看出游客；清理游客时直接 rm -rf | 需改 SaveManager；list_sessions 需合并两棵树；元数据双写 |
| 单独 `saves/_guests/<guest_id>.json` | 单文件 | 极简 | 丢失 slot/auto 机制；不能存大存档（剧情+wiki） |

**推荐：复用现有 `saves/`**。理由：
- `SaveManager.list_sessions(account_id=...)` 已经按 account_id 隔离，游客/注册用户**逻辑上**已经分开。
- `migrate_account_id` 已经能处理游客→注册用户的 owner 改写。
- 引入第二棵树会破坏这个抽象，所有 list/count/delete 都要改。

## 3. 推荐方案：A（HttpOnly 签名 Cookie） + 可选 B（恢复码） 二选一

### 推荐路径 A（最简、最稳）

**核心理由**：
1. **v2.7 仍是 beta**，先把"换浏览器不丢"做掉就够用
2. 前端**零代码改动**——`account.ts` 里 `getCurrentAccountId()` 优先读 cookie，读不到才回退 localStorage
3. 后端改动隔离在 `handler_base.py`（中间件）和 `account_system.create_guest`（签发）两个地方
4. 恢复码作为 v2.8+ 增量功能，需要时再加

**但有个真问题**需要回答：**cookie 真的能让"换浏览器"工作吗？**

答：**不能**。Cookie 跟着浏览器走，换浏览器就丢。**真正的跨设备永远需要"账号登录态"**（用户名+密码或恢复码）。

如果用户的核心诉求是"换浏览器不丢"，A 就够（cookie 走 localStorage 的备份）。
如果用户的核心诉求是"换手机/电脑也能继续"，**必须 B 或 C**。

## 4. 详细设计（方案 A：HttpOnly Cookie）

### 4.1 配置

新增环境变量：
```
WEB_COOKIE_SECRET=<random 32 bytes>   # HMAC 签名密钥
WEB_COOKIE_NAME=hfe_guest             # 默认 cookie 名
WEB_COOKIE_MAX_AGE=31536000           # 1 年
WEB_COOKIE_SECURE=true                # HTTPS only
```

`config.py` 读取到 `Server` 类下，缺省时**降级到非签名模式**（仅 base64，不防篡改；本地开发用）。

### 4.2 Cookie 格式

`hfe_guest = base64url(guest_id) + "." + base64url(hmac_sha256(guest_id, secret))`

示例：`guest_3cdb9ab1.MTIzNDU2Nzg5MGFiY2RlZg`

校验失败 → 当作新游客，触发 create_guest。

### 4.3 中间件（`web_server/handler_base.py`）

新增 `HandlerBaseMixin.guest_id` 属性：
```python
def _get_or_create_guest_id(self) -> str:
    """从 cookie 解析 guest_id；无效则创建并 Set-Cookie"""
    cookie = self._read_cookie(SERVER_COOKIE_NAME)
    raw = _verify_signed_cookie(cookie)
    if raw and raw.startswith("guest_"):
        return raw
    # 创建新 guest
    sys_inst = _get_account_system_local()
    guest = sys_inst.create_guest()
    self._set_signed_cookie(SERVER_COOKIE_NAME, guest.account_id, max_age=...)
    return guest.account_id
```

`do_GET` / `do_POST` 入口**不**自动调用这个方法（避免污染无状态接口如 `/api/eras`）。需要身份感知的路由（`/api/menu`、`/api/archives`、`/api/saves/list`、`/api/start`、`/api/account/register`、`/api/account/login`）显式调 `_get_or_create_guest_id()`。

### 4.4 路由改造

- `GET /api/menu`：移除 query `account_id` 解析，**改用 cookie**（向后兼容：cookie 没有就 query 兜底）
- `GET /api/saves/list`：同上
- `POST /api/start`：移除 body `account_id`，用 cookie
- `POST /api/account/register`：用 cookie 拿 `migrate_from_guest_id`（前端不必再带）
- `POST /api/account/login`：成功后**迁移 cookie**——如果当前是游客，迁移后再清 cookie 的标记

### 4.5 前端改造

`account.ts`：
```ts
export function getCurrentAccountId(): string | null {
  if (typeof window === 'undefined') return null;
  return localStorage.getItem(SESSION_KEY);
}
// 不变——localStorage 仍可作前端 cache；cookie 是兜底真相
```

`account.ts` 仍然保留 `ensureGuestAccountId()`，但**优先级改成**：
1. 先读 localStorage
2. 调 `/api/menu`，从 `Set-Cookie` 头拿到 guest_id（后端会强制创建）
3. 写 localStorage

实际上，**前端完全不用改**——cookie 是浏览器自动带的，`credentials: 'include'` 已经在 client.ts 里了（ofetch 默认带 credentials: 'include'）。

### 4.6 不破坏向后兼容

- 老玩家已有 `localStorage` 里的 `guest_xxx` ID：第一次访问服务端时**没有**合法签名 cookie，服务端仍按 query/body 路径接受 ID（向后兼容）
- 老玩家清掉 localStorage 但有 cookie：服务端从 cookie 拿到 ID，行为一致
- 完全新玩家：服务端创建 + Set-Cookie

### 4.7 失败/降级

- 浏览器禁用 cookie → 降级到 query 路径（已有）
- SECRET 缺失（开发）→ 用固定 fallback，warning 一次
- HMAC 校验失败（cookie 篡改）→ 拒绝、重新签发（不抛 500）

### 4.8 安全考虑

| 风险 | 缓解 |
|---|---|
| Cookie 被 XSS 偷 | HttpOnly → JS 拿不到 |
| Cookie 被网络嗅探 | Secure + HTTPS（生产强制） |
| CSRF（跨站请求） | SameSite=Lax（v2.7 阶段）；SameSite=Strict + CSRF token（v2.8+） |
| 重放 | 长期 cookie + 服务端可加最近使用时间审计 |
| 爆破（猜 guest_id） | guest_id 8 hex = 32 位熵，足够强 |

### 4.9 待用户确认的关键选择

| 选项 | A1（推荐） | A2 | A3 |
|---|---|---|---|
| 有效期 | 1 年 | 30 天（更短） | 永久（max-age 5 年） |
| 注册后 | 保留 cookie（标记为 user）| 立即清除 cookie | 保留 cookie 30 天作为恢复锚 |
| 多设备 | 各设备独立 cookie | 共享 cookie（同设备不同浏览器也算） | 各设备独立 + 登录态强绑 |
| Secure flag | dev=false, prod=true | 始终 true | 始终 true + 强制 HTTPS |

## 5. 详细设计（方案 B：恢复码）

适用场景：用户**明确不要服务端记住自己**，或**要支持跨设备**。

### 5.1 数据结构

`saves/accounts/recovery_codes.json`：
```json
{
  "codes": [
    {
      "code_hash": "scrypt:16384:8:1$...$...",
      "guest_id": "guest_3cdb9ab1",
      "created_at": "...",
      "last_used_at": "...",
      "use_count": 0
    }
  ]
}
```

恢复码本身：`HF-<4位字母数字>-<4位字母数字>-<4位字母数字>` 12 字符（不含易混字符 0/O/1/I），即 `XXXX-XXXX-XXXX` 格式。

### 5.2 流程

1. 游客首次进入 → 后端生成恢复码（`create_guest` 同时绑码）
2. 弹窗展示一次："请抄下您的恢复码：HF-XXXX-XXXX-XXXX"
3. 用户可点"再展示一次"（需先输入游客密码？v2.7 阶段游客没密码 → 暂只能展示一次，或加游客密码）

### 5.3 API

| 端点 | 方法 | 说明 |
|---|---|---|
| `POST /api/account/guest/recovery_code` | POST | 重新生成（旧的失效），body `{account_id, password_or_code}` |
| `POST /api/account/recover` | POST | 输入恢复码 → 返回新游客 session（Set-Cookie） |

### 5.4 安全考虑

- 12 字符 base32 排除易混字符 = 32^12 = 1.15×10^18 空间
- 必须限速：5/min/IP
- 一次性使用？or 多次？建议**长期有效**（用户可能换多次设备）
- 恢复后**不**强制改游客 ID（保留数据连续性），只换 cookie/session

## 6. 推荐决策树

```
用户场景？
├── 只是"换浏览器不丢"  ──→  方案 A（cookie），1-2 天工作量
├── 要"换手机也能继续"  ──→  方案 B（恢复码），3-5 天工作量
└── 都要              ──→  方案 C（A+B），1 周工作量
```

**针对当前 v2.7 beta 阶段**：
- 项目还在邀请码内测，玩家数量有限
- 账户体系本身（用户名+密码）已经能"换设备"
- 真正的"游客换设备"是个相对小众诉求

**我的建议：先做 A，把"游客换浏览器不丢"做掉；B 列入 v2.8 backlog。**

---

## 7. 不做的边界

明确**不在 v2.7 持久化方案里做的**：
- 游客"找回密码"——游客本来没密码
- 跨设备实时同步（WebSocket / SSE 推送）——过重
- 第三方登录（Google/微信 OAuth）——需要外部账号
- 游客数据云备份到 S3 之类——单服务，暂不引入
- 双因素认证（2FA）——beta 阶段过重

| 方案 | 后端 | 前端 | 测试 | 总计 |
|---|---|---|---|---|
| A cookie | 4-6h（中间件+3 路由） | 1h（前端不动） | 2h（端到端） | 半天 |
| B 恢复码 | 8-10h（码生成+API+UI 后端） | 4-6h（弹窗+输入页） | 3h | 2 天 |
| C A+B | 10-12h | 6-8h | 4h | 1 周 |
