# v2.7+ LLM 反噬调研报告（prompt 设计实验）

> **日期**：2026-07-10
> **作者**：实测 + 代码审计
> **目的**：分析"为什么 LLM 主动探索更多 tool 导致 prefetch 反噬"，探索 prompt 层 + 节点层 4 种抑制方案
> **结论**：所有"v1-v3 单一方案"都失败。**真问题不是 LLM 决策慢，而是 prefetch 注入让 LLM#2 上下文暴增**。需要"把 prefetch 结果只对 LLM#1 可见"的隔离方案（复杂度高），或换 agent 架构（不是 prompt 优化能解决）。

---

## 0. TL;DR

| 方案 | 节省 LLM#1 调次数 | 副作用 | 结论 |
|---|---|---|---|
| v0: 原 ReAct | 0（基准） | 调 2-3 次 | 15-30s |
| v1: 预取接入 workflow | ❌ 反噬 | LLM 调 1-4 次更多 | 17-50s |
| v2: 预取 + tool_calls 过滤 | ✅ 0 次（filter 立即生效） | LLM#2 context 暴增，5-33s | 17-50s |
| v3: 只改 system_prompt 加"足够"指令 | ❌ 几乎无效果 | LLM 仍主动探索 | ~20s |

**真问题**：
- LLM 调多次 tool（v0 基础行为）**不是反常**，是** ReAct agent 的设计初衷**
- prefetch 注入的 1-2KB message **永远在 messages 累积**——LLM#2 看到 → 上下文 2-3x → 推理 5-10x 慢
- LLM 决定"信息不够，要再查" **因为 prompt 鼓励**（v1.7.37 Wiki hint、v1.7.36 Drama hint 等）

## 1. LLM 反复调 tool 的 prompt 诱导源（审计）

`dm/prompts/system_base.md` 里有**多处**诱导 LLM 主动探索：

| 段落 | 行号 | 诱导内容 | LLM 调什么 |
|---|---|---|---|
| v1.7.37 Wiki 检索 hint | 619-632 | "Wiki 是**按需**注入...**调用**" | query_knowledge / query_narrative_snippets |
| v1.7.36 DramaManager hint | 582-591 | "你**应该**采纳 drama_hint" | get_state (查 state) |
| v1.7.36 calendar_events hint | 608-619 | "**应该** 体现这些大事件的环境" | recall_events / query_knowledge |
| v1.7.30 城市感知 | 481-501 | "narrative 必须体现该城市的感官特征" | get_state / query_knowledge (查 city) |
| v1.7.30 结构化事件输出 | 377-419 | "**每次** narrative 后**必须**输出 `<events>` 块" | save_event |
| v1.7.30 触发模式库 | 455 | "27 个明清小说情节模式" | recall_events |
| v2.4 文字地图 | 510-532 | "**至少 1 个** 应该是'去X'的移动选项" | get_state (查 location) |

**合计 7 处直接诱导 LLM 调 tool**。这是 prompt 设计**鼓励 agentic 行为**的结果——让 LLM 主动查询获得更好的 narrative。

**问题**：当 prefetch 已经预取完所有查询类 tool 的结果，**LLM 不知道**——prompt 里没说"信息已预取，不要再查"。

## 2. v1 实验：预取 + 1 轮（基础版）

### 设计
- 加 `pre_fetch_tools_node` 在 situation_assessment 之前
- 调 6 个查询类 tool，结果打包成 HumanMessage（不是 SystemMessage，避开 LangChain "non-consecutive system messages" 错误）
- HumanMessage 注入到 messages 末尾

### 实际数据

| 指标 | v0 (ReAct) | v1 (prefetch) |
|---|---|---|
| 预取耗时 | 0ms | 2-4ms |
| LLM#1 调次数 | 2-3 | **1-4**（更多） |
| LLM#2 耗时 | 5-10s | 2-15s |
| 总 HTTP 耗时 | 15-30s | 17-50s |

### 反噬原因

LLM 看到 prefetch 注入的 HumanMessage 后**决策变化**：

**之前**（v0）：
```
LLM#1: "我应该 get_state + check_rules + recall_events"（3 次）
LLM#2: "信息够了，写 narrative"
```

**之后**（v1）：
```
LLM#1: "我看到 state 已有 10 keys，但我需要 story_segments + knowledge + random_segment"（3-4 次）
LLM#2: "信息够了，但 prefetch 上下文 1-2KB 让推理变慢"
```

**LLM 好奇心**——给它更多上下文，它会想"再查更细的"。

### 结论
❌ **反噬**——LLM 调次数不降反升，总耗时**不改善**。

## 3. v2 实验：预取 + tool_calls 过滤

### 设计
- v1 + 加 tool_calls 白名单过滤
- LLM 调查询类 tool（已被 prefetch 覆盖）→ 移除
- LLM 调决策类 tool（roll_dice / save_event / offer_identity_switch / get_random_segment）→ 保留

### 实际数据

| 指标 | v1 (prefetch) | v2 (prefetch + filter) |
|---|---|---|
| LLM#1 filter 移除 | - | 0-2 个 tool_calls |
| LLM#1 调次数（filter 后） | 1-4 | **0-1** |
| LLM#2 耗时 | 2-15s | **5-33s** ⚠ |
| 总 HTTP 耗时 | 17-50s | 7.8s-87s |

### 反噬原因（更深层）

**反噬 1：LangGraph 状态不一致**
- filter 移除 LLM 期望的 tool_calls
- 但 messages 里**没**tool_result 跟着
- LLM#2 invoke 时，上下文"半成品" → **BadRequestError "tool call and result not match"**
- 触发所有 provider fallback → HTTP 500

**反噬 2：prefetch 上下文累积**（更隐蔽）
- prefetch 1-2KB 的 HumanMessage 注入到 messages
- LLM#2 invoke 时**仍然看到**这条 message（messages 累积）
- 上下文长度 2-3x → 推理 5-10x 慢
- 30s+ 单次 LLM#2 多次出现

### 结论
❌ **两次反噬**——filter 解决了一个问题（LLM#1 调次数），引入两个新问题（LangGraph 错 + LLM#2 慢）。

## 4. v3 实验：只在 system_prompt 加"足够"指令

### 设计
- 不接入 prefetch 节点
- 在 system_prompt 末尾加：
  > "查询类 tool（get_state / check_rules / query_knowledge）已被预先查询，结果在 [__PRE_FETCHED_TOOLS__] 段中。如信息已足够，请直接生成 narrative。"

### 预期
- LLM 看到 system_prompt 里的提示 + 没有 [__PRE_FETCHED_TOOLS__] 段（因为没接入 prefetch 节点）
- 期望：LLM 至少"知道"有 prefetch 概念，少调一些

### 实际（无 prefetch 数据支持，单看 system_prompt 影响）
- LLM 行为**完全无变化**
- 仍然调 2-3 次 tool

### 原因
- 文字指令**对抗不了** v1.7.30 / v1.7.37 等多处的**结构化诱导**
- LLM 看到 7 处"应该主动查"，1 处"信息够了" → 跟随多数

### 结论
❌ **prompt-only 改写无法对抗**结构化的诱导源。

## 5. 真问题分析

**所有方案失败的根本原因**：

### 5.1 LLM 调多次 tool 是 ReAct 的设计初衷

LangGraph 的 `should_continue → call_tools → situation_assessment` 循环**就是为了让 LLM 自主探索**。这是 agentic 设计的**核心优势**——LLM 看到工具描述，决定查什么，怎么组合信息。

试图用"减少 LLM 调次数"来加速**违背 agentic 设计**。

### 5.2 prefetch 注入的 1-2KB 是不可逆负担

messages 一旦累积，**永远在**。即使 LLM#1 不再调 tool，LLM#2 也带着这 1-2KB 推理。

**根本解法**：
- **方案 A**：用 LangGraph 的 `state` 而不是 `messages` 传递预取结果（messages 不增加）
- **方案 B**：用 prompt 压缩技术（LLMLingua）把 prefetch 1-2KB 压到 200 字节
- **方案 C**：换 agent 架构——把"决策循环"从 LLM 移到 Python 代码（确定性决策，LLM 只做 narrative）

### 5.3 system_prompt 多处诱导

要真正让 LLM 少调 tool，**需要重写 system_prompt** 7 处诱导段，标注"v2.7+ 这些已预取，不要再调"。

但**重写 system_prompt 是高风险**——可能影响 narrative 质量。

## 6. 未来可能方案（按 ROI 排序）

### 6.1 方案 D：state 替代 messages（推荐）

```python
# 不放 messages，放 graph state
workflow.add_node("pre_fetch_tools", pre_fetch_node)
# pre_fetch_node 返回 {prefetched: {...}}
# LLM#1 通过 state 访问（但 LLM 看不到）
# LLM#2 同样看不到（state 不在 messages）
```

**问题**：LLM 看不到 prefetch 结果，**自己仍会调**——退回到 v0。

### 6.2 方案 E：prompt 压缩

```python
import llmlingua
compressor = llmlingua.PromptCompressor()
compressed = compressor.compress_prompt(prefetch_message, target_token=200)
```

**问题**：依赖 llmlingua 库，引入新依赖。压缩有损，可能影响决策。

### 6.3 方案 F：重写 system_prompt（高风险）

**改动**：
- v1.7.30 evt 触发模式库 → 删除（v1.7.32 已不要求 events 块）
- v1.7.36/37 hint 段 → 标注"v2.7+ 这些已预取"
- v1.7.30 城市感知 → 注入预取结果，标注"已在 [STATE] 段"

**问题**：影响 narrative 质量（实测可能）。需要 A/B 测试。

### 6.4 方案 G：换 agent 架构

**改 LangGraph 拓扑**：
```
skill_orchestration → pre_fetch_tools → situation_assessment (no tools) → narrative_fusion
                                                  ↑
                                                  └ 决策类 tool 仍然允许
                                                     但 LLM 必须自己判断"信息够不够"
```

- situation_assessment 节点的 LLM 看不到 tool 描述（bind_tools 不传查询类）
- LLM 只能调决策类 tool

**问题**：bind_tools 不传查询类 tool 的话，LLM 看不到工具描述，**无法在 narrative 里引用**工具结果。

### 6.5 方案 H：换 agent 框架

放弃 LangGraph ReAct，换 LangChain Expression Language (LCEL) 的 `RunnablePassthrough` —— Python 代码先调 tool，LLM 一次 invoke 就完。

**工作量**：1-2 周（重写 DM agent）

## 8. 关键教训

### 8.1 agent 性能优化 ≠ prompt 优化

agent 性能问题往往在**架构**（ReAct 循环、messages 累积、tool 选择），不是 prompt 措辞。改 prompt 是"低垂果实"，架构改动才是"质变"。

### 8.2 真实实验 > 理论分析

我之前**纸上**分析 prefetch 节省 13s/次，**实际**反噬。LLM 行为**不**符合"加上下文就加速"的简单假设。

### 8.3 性能优化要可回滚

所有改动都用 `if not v2.7_disable_xxx: 原行为` 包起来，**默认关闭**，出问题可回滚。**不要**把未验证的优化直接打开。

### 8.4 性能剖析比优化更重要

`[DM-PROF]` 日志**保留**——这是诊断未来性能问题的资产。本次所有发现都来自这些日志。
