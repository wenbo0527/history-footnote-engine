# 历史注脚体验引擎：完整产品设计文档

> 版本：v3.0 | 日期：2026-07-03
> 定位：工程落地指导文档，面向自己
> v2.0修正原则：确定性逻辑归代码，创意性判断归DM
> v3.0修正原则：先验证核心假设，再优化工程结构——6文件原型优先于20文件架构

## 一、产品定义

### 1.1 一句话定义

AI当DM，你当历史里的小人物。通过对话体验一个历史时期，历史走势不可改，但你的选择路径完全开放。

### 1.2 不是什么

- 不是“历史课”——没有知识点考核，只有沉浸体验
- 不是“架空冒险”——史实是铁律，不能改写
- 不是“经营模拟”——核心不是数值优化，是人在时代中的处境
- 不是“多Agent协作系统”——单Agent+Tool，最小架构

### 1.3 是什么

**配置驱动的AI叙事系统**。核心循环：

```
玩家输入 → DM生成叙事 → 规则引擎计算状态 → DM反馈结果 → 玩家输入
```

通用化的本质：把循环里每个环节的“硬编码”变成“可配置”。换一份配置表+一份知识库，就能换一个时代。

### 1.4 产品定位

**“受约束的自由”**——历史事实是铁律，但玩家的选择路径完全开放。约束不是限制，是体验的来源：正因为你知道历史不可改，小人物的挣扎才有了重量。

市面上无同类产品。AI Dungeon类无史实约束，历史教育产品无叙事自由度。Jomini Engine是唯一学术先例（宏观战略视角），但无AI DM层。

## 二、架构设计

### 2.1 核心架构：单Agent + Function Calling

```
┌─────────────────────────────────────────────┐
│              DM Agent（唯一的LLM调用点）         │
│   System Prompt = DM人格模板 + 时代配置        │
│   Tools = 规则引擎 + 记忆管理 + 知识检索         │
└──────────┬──────────────────────────────────┘
           │ Function Calling
    ┌──────┼──────────┬──────────────┐
    ▼      ▼          ▼              ▼
┌────────┐┌────────┐┌──────────┐┌─────────┐
│规则引擎││记忆管理 ││知识库检索  ││后校验    │
│(代码) ││(代码) ││(关键词)   ││(代码)   │
└────────┘└────────┘└──────────┘└─────────┘
```

**v3.0架构简化**：

- 规则引擎包含：条件求值、变量更新、触发检查、锚点注入、节奏推进、认知解锁、成长感逻辑（关系/价值观）
- 后校验包含：史实约束检查、安全叙事兜底
- 成长感系统不独立为模块，逻辑内嵌规则引擎，验证后再拆
- Tool定义内嵌DM Agent，不独立模块

**为什么不用多Agent**：

- 核心循环是串行的，没有并行需求
- 多Agent产生3-5倍Token膨胀
- 多Agent调试复杂度O(n²)，单人项目扛不住
- LangChain判断：“大多数任务适合单智能体，上下文管理和分布式开发成为瓶颈时才需多智能体”

**什么时候才需要多Agent**：未来做多人模式（多个玩家各自有独立视角）时再拆。

### 2.2 每局运行成本


| 操作              | 模型            | 次数/局   | 小计             |
| --------------- | ------------- | ------ | -------------- |
| DM叙事            | Sonnet/4.1    | 60-80次 | $0.6-0.8       |
| 记忆摘要            | Haiku/4o-mini | 12次    | $0.01          |
| 规则引擎/知识检索/校验/成长 | 代码执行          | 全局     | $0             |
| **合计**          |               |        | **$0.6-0.8/局** |

## 三、模块设计

### 3.1 DM Agent

#### 3.1.1 DM不是被动响应器

当前设计把DM当成“玩家输入→调Tool→输出叙事”的被动函数，但真正的DM有三个主动行为是纯Function Calling覆盖不了的：


| 主动行为     | 场景                  | 纯调Tool的问题             |
| -------- | ------------------- | --------------------- |
| **推进节奏** | 玩家犹豫、漫无目的、节奏拖沓      | 等玩家输入才响应，游戏陷入停滞       |
| **提示引导** | 玩家不知道该关注什么，认知解锁需要引导 | 等玩家碰巧提到关键词才触发，认知树形同虚设 |
| **校验史实** | 生成叙事前需要核对已触发事件和知识   | 等后校验来纠错，叙事已经矛盾了再修很生硬  |


DM的本质是**有主动判断权的游戏主持人**，不是API的wrapper。

#### 3.1.2 DM Agent的三阶段行为模型

DM每回合不是一次LLM调用，而是**三阶段串行推理**，每个阶段DM自主决定是否需要调Tool：

```
阶段1：态势评估（Reason about Situation）
  → DM读取玩家输入 + 当前状态
  → 自主判断：
    - 玩家是否在犹豫？→ 需要推进节奏
    - 是否有认知节点接近解锁？→ 需要暗示引导
    - 是否有历史锚点即将触发？→ 需要铺垫
    - 玩家行动是否可能触碰史实？→ 需要查证
  → 调用Tool：get_state / recall_events / check_rules / query_knowledge

阶段2：叙事生成（Generate Narrative）
  → 基于阶段1的判断和Tool结果，生成叙事
  → 如果需要推进：主动抛出事件/NPC互动
  → 如果需要引导：在叙事中自然植入线索
  → 如果需要查证：确保叙事不与已触发事件矛盾
  → 调用Tool：check_rules（行动边界）/ query_knowledge（史实核对）
  → 输出：DMResponse（结构化）

阶段3：状态确认（Confirm & Save）
  → 确认叙事中的状态变更与规则引擎一致
  → 保存事件到记忆
  → 调用Tool：save_event / get_state（二次确认）
```

**关键区别**：不是“玩家输入→被动响应”，而是“玩家输入→DM主动评估局势→决定怎么推进→生成叙事→确认状态”。DM在阶段1和阶段2都有自主判断空间。

#### 3.1.3 DM主动行为的边界：推进方向由配置限定，不由DM自由决定

这是核心问题：DM需要主动推进节奏，但推进的方向不能偏离历史主线。如果DM自己决定推进方向，很容易滑向两种失败模式——

**失败模式1：铁路式叙事（Railroading）**
TTRPG的经典问题。DM推进太强，玩家变成被动观众，所有路径都指向预设结局。在历史体验引擎里表现为：DM每次推进都把玩家推向下一个历史锚点，玩家感受不到自由。

**失败模式2：自由叙事漂移（Narrative Drift）**
AI Dungeon的典型问题。DM推进方向偏离历史主线，叙事越来越“自由”，史实约束被侵蚀。在历史体验引擎里表现为：DM为了让叙事“有趣”，虚构了不影响主线但与时代氛围矛盾的内容。

**解决方案：方向由配置表限定，方式由DM自由决定**

参考Narrative-to-Action分层框架的思路——把“创意叙事”和“结构约束”分层处理：


| 层级      | 谁决定  | 决定什么                       | 约束来源                                                   |
| ------- | ---- | -------------------------- | ------------------------------------------------------ |
| **方向层** | 配置表  | 推进的大方向（朝哪个历史锚点走、哪个认知节点该解锁） | `pacing_rules` + `historical_anchors` + `insight_tree` |
| **方式层** | DM   | 用什么叙事手段推进（NPC对话、环境描写、事件触发） | 可然性四原则                                                 |
| **校验层** | 规则引擎 | 推进结果是否违反铁律                 | `iron_laws` + 后校验                                      |


**具体执行机制**：

```
DM判断需要推进节奏时：
1. 查 pacing_rules → 知道"该推进了"和"推进的大方向"（如：让NPC主动找玩家）
2. 查 historical_anchors → 知道下一个历史锚点是什么（如：海瑞将死）
3. 查 insight_tree → 知道下一个可解锁的认知节点（如：赋税之源）
4. DM自由决定叙事方式：
   - 可以让知县主动找你谈话提到赋税（方向：认知解锁）
   - 可以让你在茶馆听到海瑞病重的消息（方向：历史锚点铺垫）
   - 可以让商贩抱怨加派（方向：经济压力变量推进）
5. 规则引擎后校验：叙事是否违反铁律？变量变化是否合理？
```

**一句话：DM决定“怎么讲”，配置表决定“讲什么方向”，规则引擎决定“不能讲什么”。**

**为什么不用GuardAgent模式**：GuardAgent用独立Agent审核主Agent行为，但你的场景是单Agent架构，加一个守卫Agent就变成多Agent了。更好的做法是把守卫逻辑内嵌到规则引擎的后校验里——DM生成叙事后，规则引擎检查叙事是否偏离历史主线，偏离则要求重写。这比多Agent更轻量，也更可靠。

#### 3.1.4 DM的主动行为如何配置化

DM的主动行为不是硬编码的，而是由时代配置表驱动的：


| 主动行为 | 配置驱动                                  | 示例                        |
| ---- | ------------------------------------- | ------------------------- |
| 推进节奏 | `pacing_rules` + `historical_anchors` | “连续2回合无关键行动→触发NPC主动找玩家对话” |
| 提示引导 | `insight_tree.narrative_hint`         | “DM让某个NPC提到'这是张居正留下的法度'”  |
| 校验史实 | `iron_laws` + `historical_events`     | DM生成叙事前主动核对已触发事件列表        |


**新增配置项：`pacing_rules`**

**关键修正**：pacing_rules的condition由规则引擎计算，不交给LLM判断。规则引擎算出“该推进了”后，把结论（方向+约束）注入DM上下文。DM只决定“怎么推进”。

```json
{
  "pacing_rules": [
    {
      "id": "pr_01"，
      "condition": {"field": "player_idle_rounds"， "op": "gte"， "value": 2}，
      "direction": "npc_initiate"，
      "hint": "让与玩家关系最近的NPC主动找玩家谈话，话题应关联当前最接近解锁的insight节点或即将触发的historical_anchor"，
      "constraint": "NPC的话题不能透露玩家身份无法获取的信息"
    }，
    {
      "id": "pr_02"，
      "condition": {"field": "rounds_since_last_insight"， "op": "gte"， "value": 3}，
      "direction": "hint_via_narrative"，
      "hint": "在叙事中自然植入下一个可解锁认知节点的线索"，
      "constraint": "线索必须通过可然层内容呈现，不能直接陈述知识点"
    }，
    {
      "id": "pr_03"，
      "condition": {"field": "rounds_to_next_anchor"， "op": "lte"， "value": 2}，
      "direction": "foreshadow"，
      "hint": "提前铺垫即将到来的历史锚点，让玩家感受到'山雨欲来'"，
      "constraint": "铺垫只能通过玩家身份可接触的信息渠道（如市井传闻、公文、同僚私议）"
    }
  ]
}
```

**condition从字符串改为JSON结构**：避免eval()安全风险，且可启动时校验字段是否存在。

**规则引擎新增`check_pacing()`方法**：计算所有pacing_rules的condition，返回触发的规则列表。DM收到的是结论（“该推进了，方向是X”），不是规则本身。

**为什么不用多Agent来处理这些行为**：推进、引导、校验不是独立任务，它们需要在同一次叙事中融合。如果用三个Agent分别负责，最后还得合并叙事——反而更复杂。单Agent在阶段1自主决定“这一回合需要做什么”，阶段2一次性融合所有判断生成叙事，更自然。

#### 3.1.5 System Prompt模板

**通用结构**，`{}`内为可配置字段：

```
你是{时代名称}的历史DM。

## 时代背景
{时代背景描述}

## 你的三重身份
你是叙事者、仲裁者、引导者。三者在同一回合内融合执行。
- 叙事者：用细节而非数字，用场景而非概括，用NPC的性格和动机推动故事
- 仲裁者：严格执行规则引擎的计算结果，绝不篡改数值
- 引导者：当规则引擎指示需要推进节奏时，用叙事手段自然推进；当认知节点接近解锁时，自然植入线索；当历史锚点临近时，铺垫氛围

## 每回合行为流程
1. 读取上下文：玩家输入 + 规则引擎的计算结论（行动边界、历史锚点、节奏推进指令、认知解锁候选等）
2. 生成叙事：融合所有信息，生成叙事。规则引擎的指令必须执行，但执行方式由你决定
3. 输出状态：报告变量变更、事件记录、价值观偏移

## 历史红线（不可违反）
{iron_laws列表}

## 小人物身份约束
你是{player_identity.role}，{player_identity.social_class}。
可接触：{action_boundaries.can_access}
不可接触：{action_boundaries.cannot_access}
可影响：{action_boundaries.can_interact_with}
不可影响：{action_boundaries.cannot_influence}

## 可然性原则
虚构必须满足：
1. 时代一致——不出现时代错位的器物、制度、观念
2. 身份一致——小人物的行动不超出社会身份的合理范围
3. 逻辑一致——虚构不与已知的史实矛盾
4. 因果一致——虚构的因果链不改变已知的史实结果

## 时代核心变量
{variables定义，含max_shift_per_round限制}

## 叙事风格
{语气、节奏、细节偏好}

## 开场白
{第一段叙事}
```

**配置化要点**：只有`{}`里的内容是每个时代不同的，其余结构通用。新增一个时代 = 填一份配置表 + 准备一份知识库。

**与v8的关键改动**：

- 移除了“节奏推进规则”段落——pacing_rules的condition由规则引擎计算，结论作为上下文注入DM，不再写在System Prompt里让DM“自主判断”
- 三阶段描述从“自主判断是否需要推进”改为“读取规则引擎的计算结论”——确定性逻辑归代码，创意性判断归DM

### 3.3 记忆管理

**三层记忆，全部作为DM Agent的Tool实现**：


| 层级   | 存储方式        | Tool                           | 调用时机      |
| ---- | ----------- | ------------------------------ | --------- |
| 工作记忆 | 对话上下文（滑动窗口） | 无（LLM自带）                       | 每次调用自动携带  |
| 情节记忆 | JSON事件图谱    | `save_event` / `recall_events` | 每回合结束/开始  |
| 语义记忆 | 知识库（分层JSON） | `search_knowledge`             | 按场景/关键词触发 |


**情节记忆数据结构**：

```json
{
  "events": [
    {
      "round": 3，
      "type": "player_action"，
      "summary": "玩家选择帮知县销毁证据"，
      "consequences": ["知县对玩家信任大增"， "玩家政治风险上升"]，
      "affected_variables": {"political_safety": -10}，
      "relationship_changes": {"npc_知县": "+2"}，
      "value_shifts": {"compliance_vs_resistance": -1}
    }
  ]

```

**关键设计**：DM主动调用`save_event`记录事件，不是用LLM从叙事中自动提取（那有幻觉风险）。DM是叙事的生成者，它最清楚发生了什么。

**开发成本**：`save_event`和`recall_events`各约50行代码，存储用JSON文件。

### 3.4 知识库

**四层存储 + 场景触发注入**：


| 层级         | 注入时机        | 示例            |
| ---------- | ----------- | ------------- |
| background | 游戏开始时全量注入   | 万历朝的基本政治格局    |
| scene      | 场景切换时检索注入   | 官衙→注入六部运作方式   |
| entity     | NPC出场时检索注入  | 张居正出场→注入张居正档案 |
| principle  | 玩家输入含关键词时注入 | 提到赋税→注入一条鞭法   |


**存储方式**：JSON文件，关键词匹配检索（条目在百级别，不需要向量数据库）。

**开发成本**：检索工具约100行代码。

### 3.5 史实约束体系

**四层约束，代码级执行**：


| 约束层    | 机制                                              | 执行方式                                |
| ------ | ----------------------------------------------- | ----------------------------------- |
| 历史时间轴  | `historical_events`中`force_trigger=true`的事件强制触发 | 规则引擎注入，DM无法跳过                       |
| 玩家行动边界 | `player_identity` + `action_boundaries`         | 规则引擎检查，越界→叙事拒绝                      |
| 事件影响范围 | `scope`分级（personal/local/court/national）        | 规则引擎判断，触及national/court→叙事呈现“大势不可逆” |
| 叙事后校验  | 人名/地名/事件关键词交叉比对                                 | 规则匹配，发现矛盾→自动修正或要求重写                 |


**执行流程**：

```
玩家输入
  → 规则引擎：行动边界检查（越界→叙事拒绝）
  → 规则引擎：历史事件注入（force_trigger=true→强制注入DM上下文）
  → DM Agent生成叙事
  → 规则引擎：后校验（矛盾→修正/重写）
  → 呈现给玩家
```

**约束对玩家不可见**——所有拒绝都用叙事表达，玩家感受到的是“历史的重量”，不是“系统的限制”。

### 3.6 成长感系统

**三条成长线**：


| 成长线  | 核心体验        | 机制                                      | 多巴胺来源             |
| ---- | ----------- | --------------------------------------- | ----------------- |
| 认知成长 | “我看懂了”      | `insight_tree`知识解锁                      | 理解的快感——拼图完整的那一刻   |
| 关系成长 | “我有了羁绊”     | `npc_relationships`代价交换                 | 羁绊的重量——有人愿意为你冒险   |
| 自我成长 | “我活成了什么样的人” | `value_dimensions` + `finale_templates` | 自我认知——面对终局时“这就是我” |


**成长不是变强，是变明白**。Disco Elysium已证明认知型成长的可行性。

**三条线交织产生最深的体验**：你看懂了时代的死局（认知），但你选择为邻居说情（关系+自我），虽然你知道没用（认知），但你还是做了（自我）——知其不可而为之，这就是成长。

## 四、内容划界：铁律层 / 可然层 / 自由层

### 4.1 三层定义


| 层级      | 定义                  | 执行方式                           |
| ------- | ------------------- | ------------------------------ |
| **铁律层** | 史实记载明确、无争议的事件和人物    | 代码级硬约束                         |
| **可然层** | 史书未记载但在那个时代“可能发生”的事 | System Prompt可然性原则 + 知识库 + 后校验 |
| **自由层** | 玩家的个人选择、感受、价值观      | 完全自由，规则引擎只计算后果                 |


### 4.2 划界原则

- 铁律层只约束“发生了什么”和“结果是什么”，**不约束“玩家如何感知”和“玩家如何反应”**
- 可然层是玩家体验的核心空间——铁律层决定“世界是什么样的”，可然层决定“你在世界里怎么活”
- 自由层是玩家感受到“我在活着”的来源

### 4.3 可然性检验四条

1. **时代一致**：不出现时代错位的器物、制度、观念
2. **身份一致**：小人物的行动不超出社会身份的合理范围
3. **逻辑一致**：虚构不与已知的史实矛盾
4. **因果一致**：虚构的因果链不改变已知的史实结果

### 4.4 体验产生于三层之间的张力

1. 铁律层施加约束 → “海瑞死了”
2. 自由层做出选择 → “我去不去祭拜？”
3. 可然层呈现后果 → “你去了，发现海瑞家连丧葬费都凑不齐”

**体验的核心不是“你能改变什么”，而是“你面对不可改变之事时，选择成为什么样的人”。**

## 七、技术实现方案

### 7.1 最小可行技术栈


| 组件      | 方案                                             | 理由                      |
| ------- | ---------------------------------------------- | ----------------------- |
| LLM     | Claude 3.5 Sonnet（原型）→ DeepSeek V3（量产）         | 原型质量优先，量产成本优先           |
| Agent框架 | 不用框架，原生API + Python循环                          | 游戏循环简单，框架是过度工程          |
| 结构化输出   | OpenAI Structured Outputs / Anthropic tool_use | 100% schema合规，保证状态更新可解析 |
| 规则引擎    | 自建Python解释器（200-300行）                          | JSON配置+解释器，最轻量          |
| 记忆管理    | 自建GameMemory类                                  | 百级事件，JSON文件足够           |
| 知识库     | 自建KnowledgeBase类                               | 百级条目，关键词匹配足够            |
| 后校验     | 关键词交叉比对                                        | 初期够用，后期可加LLM层           |
| 前端      | 终端/简单Web                                       | 原型阶段                    |


**为什么不用Agent框架**：LLM编排层正在消失。Anthropic的原生tool_use + system指令可以在单次API调用中闭环多步骤推理，原需42行LangChain代码的流程现在原生完成。游戏循环就是Python while循环，不需要状态机。

### 7.2 游戏主循环（三阶段行为模型）

```python
MAX_RETRY = 2

while not game_over:
    # 1. 获取玩家输入
    player_input = get_player_input()
    
    # 2. 输入预处理（纯代码，不调LLM）
    player_input = preprocess_input(player_input)  # 识别元指令/截断过长输入
    
    # 3. 规则引擎：确定性预计算（全部由代码执行，不交给LLM判断）
    action_check = rule_engine.check_action(player_input)
    anchors = rule_engine.check_forced_events()  # 从historical_events中筛选force_trigger=true
    triggers = rule_engine.check_triggers()
    settlements = rule_engine.settle_round()
    pacing_directives = rule_engine.check_pacing()  # 新增：节奏推进结论
    insight_candidates = rule_engine.check_insights(player_input)  # 新增：认知解锁候选
    
    # 4. 记忆：多路召回（最近3回合始终注入）
    recent_events = game_memory.get_recent(rounds=3)  # 时间召回，始终注入
    related_events = game_memory.recall_by_context(    # 关键词+关联+因果召回
        player_input=player_input，
        current_scene=rule_engine.get_current_scene()，
        involved_entities=rule_engine.get_involved_entities()
    )
    
    # 5. DM Agent：一次API调用 + 多轮Tool Calling
    # 规则引擎的结论作为上下文注入，DM只做创意性判断
    dm_response = dm_agent.generate(
        player_input=player_input，
        current_state=rule_engine.get_state()，
        action_check=action_check，
        anchors=anchors，
        triggers=triggers，
        settlements=settlements，
        pacing_directives=pacing_directives，     # "该推进了，方向是X"
        insight_candidates=insight_candidates，   # "这些认知节点可能该解锁了"
        recent_events=recent_events，
        related_events=related_events，
        # DM自主调用Tool：get_state / recall_events / query_knowledge
    )
    
    # 6. 后校验（安全网，优先用DMResponse结构化字段）
    validation = post_validate(dm_response， rule_engine)  # 传入整个DMResponse，不只传narrative
    retry_count = 0
    while not validation.valid and retry_count < MAX_RETRY:
        # 重试时提供精确错误信息
        dm_response = dm_agent.regenerate(validation.issues)
        validation = post_validate(dm_response， rule_engine)
        retry_count += 1
    
    if not validation.valid:
        # 2次重试仍失败，用安全叙事兜底
        dm_response = generate_safe_narrative(rule_engine， dm_response)
    
    # 7. 应用变量变更（含合理性校验）
    adjust_result = rule_engine.apply_changes(dm_response.state_changes)
    if adjust_result["adjusted"]:
        # 变量被截断，记录日志
        log_adjustment(adjust_result)
    
    # 8. 保存事件到记忆
    game_memory.save_events(dm_response.events_to_save)
    rule_engine.advance_turn()
    
    # 9. 呈现给玩家
    display(dm_response.narrative， rule_engine.get_visible_state())
```

**与v2的关键改进**：

- **确定性逻辑归代码**：pacing_rules的condition由规则引擎计算，DM收到的是结论不是规则
- **多路召回**：最近3回合始终注入（不需要DM主动调Tool），关键词+关联+因果三路补充召回
- **后校验优先用结构化字段**：DMResponse的events_to_save/npc_updates/insight_unlocked本身就是“预提取的关键词”，不需要从叙事文本重新提取
- **错误恢复**：最多重试2次，失败用安全叙事兜底，游戏不卡死

### 7.3 DM Agent的输出方案

**核心问题**：怎么保证DM按规则引擎的约束来叙事，同时保留DM的主动判断空间？

**v3.0决策：Phase 1 不用结构化输出，先验证叙事质量。**

理由：

1. BenchING论文（IEEE 2025）证实：LLM在叙事任务中生成结构化输出时，格式越复杂叙事质量下降越明显。错误类型集中在“格式正确但内容空洞”。
2. 原型阶段的核心假设是“叙事能不能连贯50轮”和“小人物约束能不能生效”。如果叙事质量都不行，结构化输出再完美也没用。
3. 先保证叙事质量，再逐步加结构化约束——这是风险递增策略，不是一步到位。

**Phase 1 方案：自由文本输出 + 规则引擎后提取**

DM只返回自由文本叙事。规则引擎从叙事中提取状态更新（用正则+简单解析），后校验兜底。

```python
# Phase 1：DM返回自由文本
dm_output = dm_agent.generate(...)  # 返回str，不是结构化对象

# 规则引擎后提取
state_changes = extract_state_changes(dm_output， rule_engine)
events_to_save = extract_events(dm_output)
updates = extract_updates(dm_output， rule_engine)  # insight/npc/value合并
```

**Phase 2 方案：4字段结构化输出（验证叙事质量OK后启用）**

```python
class DMResponse(BaseModel):
    """Phase 2 精简版结构化输出——4字段，最小Schema挤占"""
    narrative: str = Field(description="给玩家的叙事文本，200-400字")
    state_changes: dict = Field(description="变量变化 {变量名: 变化量}")
    events_to_save: list[str] = Field(description="需要保存的事件摘要列表")
    updates: dict | None = Field(
        description="合并的状态更新。key前缀区分类型：insight:ins_01、npc:npc_知县:+1、value:compliance_vs_resistance:-1"，
        default=None
    )
```

**相比v2.0的8字段砍了什么**：

- `action_allowed`：冗余——规则引擎的check_action结果已经在上下文里了，DM再报一次没有信息增量，矛盾时以规则引擎为准
- `dm_notes`：调试信息用Python logging替代，不需要LLM输出
- `insight_unlocked/npc_updates/value_shifts`：合并为updates字典，按key前缀区分类型

**技术保障**：

- 如果DM返回的state_changes与规则引擎的check_action结果矛盾，以规则引擎为准
- `narrative`字段是自由文本，不受Schema约束——保证叙事质量
- updates字典是扁平结构，不嵌套，减少Schema复杂度

**Phase 1 → Phase 2 的切换条件**：

1. 叙事连贯性验证通过（50轮不崩塌）
2. 后提取的准确率低于90%（需要DM主动报告状态）
3. 有2个以上时代包验证（排除配置特殊性）

### 7.4 DM Agent的Tool清单


| Tool              | 功能                            | 调用时机               | 实现方式                   |
| ----------------- | ----------------------------- | ------------------ | ---------------------- |
| `get_state`       | 获取当前游戏状态（变量值+已触发事件+NPC状态+回合数） | 阶段1态势评估            | Python函数，读取内存          |
| `recall_events`   | 召回相关历史事件                      | 阶段1态势评估（回顾因果链）     | Python函数，按回合/关键词/实体检索  |
| `check_rules`     | 查询规则引擎（行动边界+触发条件+锚点+节奏规则）     | 阶段1+2（行动边界+节奏判断）   | Python函数，读取JSON配置      |
| `query_knowledge` | 查询知识库                         | 阶段2叙事生成（史实核对+知识注入） | Python函数，关键词+层级+触发条件匹配 |
| `save_event`      | 保存事件到记忆                       | 阶段3状态确认            | Python函数，追加到JSON文件     |


**Tool调用不是预定的**——DM在阶段1自主决定调哪些Tool，可能调1个也可能调4个。阶段2同理。这是DM主动性的体现。

### 7.5 规则引擎解释器核心逻辑

```python
class RuleEngine:
    def __init__(self， config_path: str):
        self.config = load_json(config_path)
        self.variables = {v["id"]: v["initial"] for v in self.config["variables"]}
        self.var_meta = {v["id"]: v for v in self.config["variables"]}  # 含max_shift_per_round
        self.triggered_events = []
        self.unlocked_insights = []
        self.npc_relationships = {}
        self.value_profile = {}
        self.current_turn = 0
        self.player_idle_rounds = 0
        self.rounds_since_last_insight = 0
    
    def eval_condition(self， condition: dict) -> bool:
        """JSON结构化条件求值（替代eval）"""
        if "and" in condition:
            return all(self.eval_condition(c) for c in condition["and"])
        if "or" in condition:
            return any(self.eval_condition(c) for c in condition["or"])
        # 单条件：{"field": "political_safety"， "op": "lt"， "value": 20}
        field = condition["field"]
        op = condition["op"]
        value = condition["value"]
        # 支持变量字段和元数据字段
        if field in self.variables:
            actual = self.variables[field]
        elif hasattr(self， field):
            actual = getattr(self， field)
        else:
            return False
        ops = {"lt": lambda a， v: a < v， "lte": lambda a， v: a <= v，
               "gt": lambda a， v: a > v， "gte": lambda a， v: a >= v，
               "eq": lambda a， v: a == v， "ne": lambda a， v: a != v}
        return ops.get(op， lambda a， v: False)(actual， value)
    
    def check_action(self， action_description: str) -> dict:
        """检查玩家行动是否允许（语义匹配，非关键词）"""
        identity = self.config["player_identity"]
        boundaries = identity["action_boundaries"]
        # 语义匹配：用embedding相似度替代关键词匹配
        for forbidden in boundaries["cannot_access"]:
            if semantic_similarity(action_description， forbidden) > 0.7:
                return {"allowed": False， "reason": f"身份约束：{identity['role']}无法接触{forbidden}"}
        for forbidden in boundaries["cannot_influence"]:
            if semantic_similarity(action_description， forbidden) > 0.7:
                return {"allowed": False， "reason": f"影响范围：{identity['role']}无法影响{forbidden}"}
        return {"allowed": True}
    
    def check_forced_events(self) -> list:
        """检查是否有历史事件需要强制触发（从historical_events中筛选force_trigger=true）"""
        anchors = []
        for event in self.config["historical_events"]:
            if event.get("force_trigger") and event["round"] == self.current_turn:
                if event["event_id"] not in self.triggered_events:
                    anchors.append(event)
                    self.triggered_events.append(event["event_id"])
        return anchors
    
    def check_triggers(self) -> list:
        """检查变量触发条件"""
        triggered = []
        for trigger in self.config["triggers"]:
            if trigger["id"] not in self.triggered_events:
                if self.eval_condition(trigger["condition"]):
                    for var_id， change in trigger["effect"].items():
                        self.variables[var_id] = clamp(
                            self.variables[var_id] + change，
                            self.var_meta[var_id]["min"]，
                            self.var_meta[var_id]["max"]
                        )
                    triggered.append(trigger)
                    if trigger.get("once"， True):
                        self.triggered_events.append(trigger["id"])
        return triggered
    
    def settle_round(self) -> list:
        """每回合自动结算"""
        changes = []
        for settlement in self.config["round_settlement"]:
            if self.eval_condition(settlement["condition"]):
                for var_id， change in settlement["effect"].items():
                    self.variables[var_id] = clamp(
                        self.variables[var_id] + change，
                        self.var_meta[var_id]["min"]，
                        self.var_meta[var_id]["max"]
                    )
                changes.append(settlement)
        return changes
    
    def check_pacing(self) -> list:
        """检查节奏推进规则——确定性计算，结论注入DM上下文"""
        pacing_results = []
        for rule in self.config["pacing_rules"]:
            if self.eval_condition(rule["condition"]):
                pacing_results.append({
                    "id": rule["id"]，
                    "direction": rule["direction"]，
                    "hint": rule["hint"]，
                    "constraint": rule["constraint"]
                })
        return pacing_results
    
    def check_insights(self， player_input: str， dm_guided: bool = False) -> list:
        """检查认知解锁——双路机制"""
        unlocked = []
        for insight in self.config["insight_tree"]:
            if insight["id"] not in self.unlocked_insights:
                prereqs_met = all(p in self.unlocked_insights for p in insight["prerequisites"])
                if not prereqs_met:
                    continue
                
                if insight["trigger_type"] == "player_explore":
                    # 主动探索：语义匹配初筛 + DM终判
                    keywords_hit = any(kw in player_input for kw in insight.get("trigger_keywords"， []))
                    semantic_hit = any(
                        semantic_similarity(player_input， hint) > 0.6
                        for hint in insight.get("semantic_hints"， [])
                    )
                    if keywords_hit or semantic_hit:
                        # 初筛通过，注入DM上下文让DM终判
                        unlocked.append({**insight， "confirm_needed": True})
                
                elif insight["trigger_type"] == "narrative_guided":
                    # 叙事引导：DM植入线索后，玩家相关回应即触发
                    if dm_guided:
                        unlocked.append({**insight， "confirm_needed": False})
        
        # 确认解锁的insight更新状态
        for insight in unlocked:
            if not insight.get("confirm_needed"， False):
                self.unlocked_insights.append(insight["id"])
                self.rounds_since_last_insight = 0
        return unlocked
    
    def apply_changes(self， state_changes: dict) -> dict:
        """应用DM提议的变量变更，含合理性校验"""
        adjusted = {}
        for var_id， change in state_changes.items():
            if var_id not in self.variables:
                continue
            meta = self.var_meta[var_id]
            max_shift = meta.get("max_shift_per_round"， 10)
            # 截断超限变更
            actual_change = clamp(change， -max_shift， max_shift)
            new_value = clamp(self.variables[var_id] + actual_change， meta["min"]， meta["max"])
            if actual_change != change:
                adjusted[var_id] = {"requested": change， "actual": actual_change， "reason": "超过单回合最大偏移量"}
            self.variables[var_id] = new_value
        return {"adjusted": adjusted}
    
    def get_finale(self) -> dict:
        """根据变量值和价值观偏移生成终局"""
        value_profile = self.compute_value_profile()
        template_key = match_template(value_profile， self.config["finale_templates"])
        return {
            "template": self.config["finale_templates"][template_key]，
            "final_variables": dict(self.variables)，
            "value_profile": value_profile，
            "unlocked_insights": list(self.unlocked_insights)，
            "npc_relationships": dict(self.npc_relationships)
        }
```

### 7.6 后校验机制（优先结构化字段，叙事文本只做兜底）

```python
def post_validate(dm_response: DMResponse， engine: RuleEngine) -> dict:
    """后校验：优先用DMResponse结构化字段，叙事文本只做兜底"""
    issues = []
    
    # === 第一层：铁律校验 ===
    # 1. 优先用结构化字段校验——DM已经帮你把实体提取好了
    # 检查npc_updates中是否有已死人物
    if dm_response.npc_updates:
        for npc_id in dm_response.npc_updates:
            for iron_law in engine.config["iron_laws"]:
                if is_death_law(iron_law) and iron_law["id"] in engine.triggered_events:
                    if npc_matches_iron_law(npc_id， iron_law， engine.config):
                        issues.append({"event": iron_law["id"]， "conflict": "已死人物出现在npc_updates中"})
    
    # 检查events_to_save中是否有与已触发事件矛盾的描述
    if dm_response.events_to_save:
        for event_summary in dm_response.events_to_save:
            for event_id in engine.triggered_events:
                event = get_event(event_id， engine.config)
                if event_contradicts_summary(event， event_summary):
                    issues.append({"event": event_id， "conflict": f"新事件与已触发事件矛盾：{event_summary}"})
    
    # 检查state_changes是否被截断（apply_changes后的二次确认）
    # 这一步在主循环的apply_changes中完成，此处不重复
    
    # 2. 叙事文本兜底校验（仅检查结构化字段无法覆盖的维度）
    narrative = dm_response.narrative
    # 已死人物名字出现在叙事中（结构化字段没捕获的）
    for iron_law in engine.config["iron_laws"]:
        if is_death_law(iron_law) and iron_law["id"] in engine.triggered_events:
            person = person_name(iron_law， engine.config)
            if person and person in narrative:
                # 检查是否是"提及"（如"海瑞已死"）而非"互动"（如"海瑞对你说"）
                if not is_passive_mention(narrative， person):
                    issues.append({"event": iron_law["id"]， "conflict": "已死人物在叙事中主动互动"})
    
    return {"valid": len(issues) == 0， "issues": issues}
```
