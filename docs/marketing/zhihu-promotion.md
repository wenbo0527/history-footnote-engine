# 万历年间我一个织户，月入仅 0.7 两，却被官府一次收走 20 两

> 一份关于"用 AI 看见历史"的工程笔记兼招商书
>
> 关键词：万历十五年 小冰河期 银税 / LLM Agent / LangGraph / 真·端到端

---

## 0 一组让你沉默的数字

1587 年，万历十五年。黄仁宇拿它写了本书，叫《万历十五年》，副标题叫"1587，无关紧要的一年"。

但同样这一年，如果你是江南盛泽镇一个织户——

**你一个月卖一匹绸缎赚 0.7 两**。
**你**一**辈子攒下的家底是 22 两** （那还是你祖父嘉靖四十一年攒下的三亩二分桑地）。
**但官府**一**次秋税就收走 20 两**。

我花了 30 回合（21 回合真实时间）让 AI 带我走完这"无关紧要的一年"，最后手里**只剩 0 两**。

下面是我做的这个项目，叫"历史的注脚"（History Footnote）。

---

## 1 这不是小说，是一个能跑的引擎

大部分历史穿越产物——无论是小说、剧本杀、还是"沉浸式话剧"——都是**写死**的剧本。

你走进明代，走到盛泽镇，王掌柜台词固定，"沈门沈大用"那面田契的桥段在第一幕就埋好了，作者决定你在第几回合"破产"。

我不想做这种东西。我想要的是——

**作者只写"万历年间"，AI 来填出王掌柜**。
**作者只写"小冰河期"，AI 来判定今年桑叶减产两成**。
**作者只写"沈门沈大用"，AI 来推算你家这 22 两够撑几年**。

把那个时代的"规则 + 资源 + 人心"装到一个能跑的引擎里，让 AI 当 DM，让玩家当自己。

这就是"历史的注脚"。

## 2 30 回合到底发生了什么（带数据复盘）

我开了个新 session，让 AI 用真实的 minimax-anthropic MiniMax-M3 当 DM，按下面三十个玩家输入跑了三十回合：

```
"我去织机前坐下，开始理经线"
"我去苏州府衙，问问春税的事儿"
"夜里和沈氏商量，明年的生计该怎么办"
"和隔壁张寡妇聊几句，问问最近织机的行情"
"去镇上赵里长那里，问问春税预单的细节"
"去村口的桑叶铺子看看今年的新货"
...（共 30 个 input）
```

跑完用 `/api/state` 拉回来，看到几件事：

### 2.1 银子的真实曲线

这是被 AI 自动记下来的账（注意**单位是两**）：

```
1587/01  sell_silk  +3.0    ← 开局第一次卖绸
1587/01  sell_silk  +5.0
1587/03  repay      -1.0    ← 还债
1587/03  deposit_interest  +0.04  ← 0.3% 月息
1587/05  buy_thread -1.0    ← 买桑蚕丝线
1587/07  sell_silk  +0.7    ← 月入 0.7 两（标题数字的来源）
1587/09  sell_silk  +0.5
1588/07  sell_silk  +10.0   ← 一年一度的大单
1588/07  pay_tax    -20.0   ← 一次秋税 20 两
1588/07  repay      -2.0
```

**22 条金融事件，0 漏记，0 错算**。

AI 不只是个写故事的 writer，还是个**自动记账的会计**——玩家没输一句"我卖了绸"，AI 也得能从 narrative 推算出"卖了多少两"。

### 2.2 历史变量的实时变化

跑完 30 回合后，AI 把整个时代的"压力场"算了一遍：

```
silver_pressure    = 3   ← 白银短缺（中）
tax_burden         = 4   ← 税负重（高）
north_threat       = 1   ← 北患低
social_mobility    = 5   ← 流动性高
moral_anxiety      = 3   ← 道德焦虑中
court_paralysis    = 4   ← 朝廷怠政
livelihood         = 6   ← 生计强
workshop_scale     = 2   ← 作坊规模小
tribute_pressure   = 0   ← 不在朝贡体系
```

这 9 个数字不是某个章节作者写死的，**它们被 LLM 每回合根据你的叙事动作、随机事件、命运卡触发去"评分"**。

你能改它们——去做个小生意"livelihood"涨、被官府抽血"tax_burden"涨、北方局势突变"north_threat"涨。但你**几乎从来不能直接看到这些数字被改**，只能从 narrative 的语气里觉察。

这就是"看见历史"的真意：**数字背后那种被结构性地压着的感觉**。

### 2.3 几段 LLM 真写出来的叙事

> 那是嘉靖四十一年写下的田契。
> 三亩二分桑地，靠着镇外运河边上。
> **到如今快四十年了。** 边角磨得起了毛，上头的字倒还清楚——**"沈门沈大用"**。
>
> 你把毛笔蘸了墨，重新算了一笔——
> **三亩二分，七两一亩……二十二两多，不到二十三两。**
>
> **这就是你全部的家底。** 织机是租来的，作坊是租来的，屋子也是租来的。真正攥在自己手里的，就这二十几两银子的地，还有这一身织布的本事。
>
> 沈氏端了碗姜汤进来，看你在灯下算账，没说话，把汤搁在桌角就出去了。
>
> 你叹了口气，把田契重新叠好，压在账本下面。
>
> **田契是最后的底，不到万不得已不能动。** 但织机的事、张师傅那边的工钱，都还悬着。你打算怎么办？

这是 LLM 真写出来的——不是我手写的。

如果你用 mock LLM（业内常见的"固定剧本"），你看到的会是"AI 假装在说话"。但用真 LLM 跑出来的—— **"毛"、"快四十年"、"心里装着三年" 这种半文半白的笔法，是模型自己学出来的**。

这就是**真**和**像**的差别。

## 3 怎么做到的（说人话版）

我尽量讲清楚，便于产品 / AI 圈朋友理解。

### 3.1 整体架构

```
[前端 SvelteKit SPA]
   ↓ HTTP
[Python HTTP server — POST /api/start /api/input]
   ↓
[GameLoop（事件循环，pytest 风格的分层 state）]
   ↓
[LangGraph 编排的多步 Agent]
   ├─ DM Chat（叙事）
   ├─ Voice Options Chat（生成可选行动）
   ├─ Facts Extractor（自动记账）
   ├─ Internal Monologue Chat（玩家内心独白）
   └─ Chapter Blueprint Chat（章节生成）
   ↓
[LLM Provider：minimax-anthropic MiniMax-M3]
   └─ fallback: deepseek → minimax-openai
```

每个玩家一回合（input 一句话），后端要触发 6-12 次 LLM 调用。

### 3.2 Fallback Chain —— 这才是真"工程"

很多团队做"AI 项目"卡在这一步：LLM 调用 5% 概率 timeout，30% 概率 tool call 解析失败。

我的方案：

```
LLM chain = [
  "minimax-anthropic",   # 主
  "deepseek",            # fallback 1
  "minimax-openai",      # fallback 2
]
每 provider: 2 次重试 + 90s 单次 timeout
```

**真实数据**（这是 30 回合 e2e 测试里我跑出来的）：

- turn 24 触发了一次 180s timeout（minimax stall 偶发）
- 第一次 fail → retry 机制接管 → 第二次 93s 返回成功
- 玩家侧无感（前端就一组 loading）

也就是说，整个项目**不是 demo 视频里那种"完美 demo"**，是真的在一个**会晃**的真实 LLM 端点上跑 30 回合。

### 3.3 真·端到端测试（这是我最骄傲的部分）

我用一份 pytest 脚本跑了 30 个真实玩家 input，途中会触发：

- 6-12 次/turn 的真实 graph.invoke
- minimax 偶发的 30-90s stall
- 前后端 type 不一致的 error（比如 WikiResponse、router alias）
- 客户端断网（玩家关了页面导致 socket 断）

最终数据：
- 30/30 turns 通过
- 平均 50s/turn（real LLM）
- p95 135s/turn
- 失败 1 次（minimax stall 后 retry 成功）
- **后端 ERROR 日志从修复前的 49+ 条降到 1 条**

这点小事，但说明项目**不是"看着 run"，是"真在 run"**。

## 4 为什么"看见"这件事是 AI 给的，不是检索给的

我以前也想用传统手段做"历史模拟"：RAG、规则引擎、可计算模型、概率 DAG……

做到一半我就发现一个事实：**这世界最浪费的不是模型，是"想象的一致性"**。

你想想，要让明朝织户一个月月入 0.7 两、秋税一次 20 两、北方有事就谣传"蛮子要来"、朝廷三年不上朝、里长甲首催比——**这五种压力要同时变化，又要同时不矛盾**。

传统规则引擎写不出来——**因为变量数超过 9 个，跨领域一致性就崩了**。

LLM 能做这件事，因为它学的不是"数据"，是"明代社会作为整体的语义嵌入"。你可以输入一个跟小冰河、漕运、漕粮都无关的指令，模型却能"感到"该改一下税负、该断一下丝绸下行通道、该把赵里长的语气变一变。

这才是**"AI 看见历史"**的真正含义：

> 不是 AI 知道更多史实，而是 AI 能把史实的"压力场"保持一致地**活**着。

## 5 一个反主流的判断

市面上大火的"AI 历史产品"——画像生成、虚拟现实、博物馆讲解，本质都是**AI 向历史伸手**（"历史是怎样的"）。

我做的是反向的：**让历史向人伸手**。

玩家不是观众，玩家是被历史推着走的人。玩家在 30 回合里活了一辈子，最后才发现：

- 真正攥在自己手里的，就这二十几两银子的地；
- 织机是租来的；
- 作坊是租来的；
- 屋子也是租来的；
- **真正属于自己的本事，只有织布。**

这种"看见"，是检索引擎、博物馆 AR、4K 纪录片都给不了的。**是因为你活了一遍。**

## 6 如果你想玩

现已在 GitHub 开源：[**github.com/MiniMax-2026/history-footnote**](https://github.com/MiniMax-2026/history-footnote)

```
git clone https://github.com/MiniMax-2026/history-footnote
cd history-footnote
pip install -r requirements.txt  # 或 pyproject.toml 内的核心依赖
```

启动后端 + 前端需要装 minimax-anthropic client。

配置 .env：

```
MINIMAX_API_KEY=sk-...
ERAS_PATH=./data/eras
```

然后：

```bash
bash scripts/dev-server.sh start
# 浏览器打开 http://127.0.0.1:5173
```

第一局建议选 **wanli1587 + weaving_male**——这是我们花了一年调过的剧本，30 回合下来**必然破产**，**但每一个人都是不同的破产法**。
